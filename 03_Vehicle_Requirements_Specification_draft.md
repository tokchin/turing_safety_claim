# 車両要求仕様書（Vehicle Requirements Specification）骨子ドラフト

本書は、Turing_Safety_Claim.md の衛星文書として位置付ける、L4対応車両プラットフォームに対する要求仕様の骨子である。

本書の目的は、TuringのADS（通常走行用ECU + MRC用ECU の異種ECU冗長構成）を搭載するL4対応車両として、OEMが提供すべき車両プラットフォームの要件を、量産設計判断に耐える定量レベルで提示することである。

本書は、Turing Safety Claim 第10章（センサ・計算機・車両制御系の失陥安全）が示す失陥モードと挙動原則を、量産設計に落とし込む際の要求として用いる。

---

# 1. 本書の位置付け

## 1.1 上位文書との関係

本書は、Turing_Safety_Claim 第2版の以下の章を実装側に展開する文書である。第10章（失陥安全）／第11章（MRC設計）／第14章（サイバーセキュリティ方針）／第17章（責任分界・技術）／Appendix I（責任分界マトリクス）。

VMC I/F仕様書（02_VMC_IF_Specification_draft.md）と相互補完する。本書は車両プラットフォーム側の冗長性・性能・品質要求を扱い、VMC I/F仕様書はI/Fプロトコル・指令フォーマットを扱う。

## 1.2 本書の確定スコープ

本書では、L4対応車両として満たすべき制動・操舵・電源・通信・センサ取付・HMI・サイバーセキュリティの定量要求を確定する。

## 1.3 本書の確定外スコープ

ベース車両の選定、車体構造、内装、外装、エネルギーマネジメント詳細、サプライヤ固有の実装は、本書では確定しない。これらはOEMとサプライヤが本書要求を満たす形で確定する。

## 1.4 数値の暫定性

本書に記載する数値は暫定値であり、HARA結果、ODD定義書（お台場ODD）、検証エビデンスを踏まえて確定する。確定値は、本書改版または別途協定書で管理する。

# 2. 全体方針

## 2.1 L4対応車両の基本要件

本書が対象とする車両は、Turing_Safety_Claim の最上位安全主張、すなわち「一系統失陥時に車両単独でリスク最小状態に到達する」ことを物理的に支える冗長性を備えた車両プラットフォームである。

具体的には、ADSが片系喪失した状態でも、もう一方の系統からのMRC指令（最大減速、操舵、駆動遮断、ハザード、停止保持）を確実に実行できる車両である。

## 2.2 Fail-Operational vs. Fail-Safe の方針

本書では、Safety-Critical な車両機能について、Fail-Operational（片系喪失でも機能維持）を求める。具体的には、制動、操舵、電源、通信ネットワーク、車両状態センサ。

Fail-Safe（片系喪失で安全側に縮退）で許容する機能は、駆動（駆動遮断はFail-Safeで足りる）、灯火・HMI（一部）。

## 2.3 ASIL目標

各機能のASIL目標は、HARA結果に基づき確定する。本書暫定値は次の通り。

| 機能 | ASIL目標（暫定） |
| --- | --- |
| 制動 | ASIL-D |
| 操舵 | ASIL-D |
| 駆動遮断 | ASIL-C |
| 駆動指令応答 | ASIL-C |
| 電源（LV1/LV2） | ASIL-D |
| バックボーン通信 | ASIL-D |
| 灯火・ハザード | ASIL-B |
| 外向けHMI | ASIL-B |
| ECU健全性監視 | ASIL-D |

# 3. 制動系冗長要件

## 3.1 アーキテクチャ要求

制動系は、片系喪失時にも単独で必要減速度を発揮できるDual-channel冗長アーキテクチャとする。

候補アーキテクチャは次のいずれか以上の冗長性を満たすこと。

| 構成 | 概要 |
| --- | --- |
| Dual hydraulic | プライマリ油圧系統 + バックアップ油圧系統 |
| Brake-by-wire dual channel | 電動キャリパ + バックアップアクチュエータ（油圧 or 電動）|
| Hybrid（推奨） | プライマリBrake-by-wire + バックアップ油圧 |

## 3.2 性能要求（暫定値）

| 項目 | 要求値（暫定） |
| --- | --- |
| 通常時最大減速度 | ≥ 9.0 m/s² |
| 片系喪失時の保証減速度 | ≥ 6.0 m/s² |
| MRC時の標準減速度 | 3.0〜5.0 m/s²（路面・速度・後続車有無で調整） |
| 緊急時最大減速度（片系喪失時） | ≥ 5.5 m/s² |
| 制動応答遅延（指令→制動開始） | ≤ 50ms |
| Fault Tolerant Time Interval（FTTI） | ≤ 100ms |

具体値は、お台場ODDでの停止距離要求およびHARA結果で確定する。

## 3.3 故障モードと挙動

| 故障モード | 要求挙動 |
| --- | --- |
| プライマリ油圧/電動失陥 | バックアップ系統で減速度6.0 m/s²以上を維持、ADSへ通知、Plausibility失敗を申告 |
| バックアップ系統失陥 | プライマリ単独で動作、ADSへ通知、MRC遷移要因として申告 |
| 両系喪失 | 駐車ブレーキで補助、駆動遮断、ADSへ最重大故障通知 |
| 制動指令断 | 直近指令の保持時間（≤ 200ms）後、自律的に保守的減速 |
| ABSセンサ失陥 | ABS無効化通知、ADS側で予備的減速度設定 |

## 3.4 単一電源喪失への許容性

LV1電源喪失時にも、LV2電源で制動機能の少なくとも一系統が維持されること。電源系統の独立性要件は第6章を参照。

## 3.5 駐車ブレーキ要件

電動駐車ブレーキ（EPB）は、停止保持の主機能として用いる。MRC完了後の停止保持中、消費電力を最小化しつつ意図せざる動き出しを防止する。

EPBは、Dual actuator（左右独立）で構成し、片側失陥時にも停止保持能力を維持する。

# 4. 操舵系冗長要件

## 4.1 アーキテクチャ要求

操舵系は、Dual-wound EPS（または同等以上の冗長EPS）を前提とする。具体的には、コイル巻線、インバータ、ECU、ポジションセンサを冗長化する。

操舵Actuatorのトルク容量は、片系単独でMRCに必要な操舵性能を発揮できること。

## 4.2 性能要求（暫定値）

| 項目 | 要求値（暫定） |
| --- | --- |
| 通常時最大舵角速度 | ≥ 500 deg/sec |
| 片系喪失時の保証舵角速度 | ≥ 250 deg/sec |
| 通常時最大トルク | ≥ 12 Nm |
| 片系喪失時の保証トルク | ≥ 7 Nm |
| センター戻し性能（片系喪失時） | センター戻り遅延 ≤ 1.5sec |
| 操舵応答遅延（指令→Actuator動作） | ≤ 30ms |
| FTTI | ≤ 100ms |

## 4.3 操舵角センサ冗長

Steering Angle Sensorは、Dual sensor構成とする。両センサの差分監視により、片系故障を検知する。差分が許容値（暫定 1.0 deg）を超えた場合、ADSに健全性低下を申告する。

## 4.4 故障モードと挙動

| 故障モード | 要求挙動 |
| --- | --- |
| 片系巻線/ECU失陥 | 残存系で性能維持、ADSへ通知、MRC遷移要因として申告 |
| 両系喪失 | 操舵フリー（または最後の指令を保持しつつフェイルセーフ）、ADSへ最重大故障通知 |
| 操舵指令断 | 直近指令を保持（≤ 200ms）、その後センター戻し |
| Steering Angle Sensor片系失陥 | 残存センサで動作、Plausibility監視を強化、ADSへ通知 |

## 4.5 物理的セパレーション

冗長要素（コイル、インバータ、ECU）は、共通原因故障を避けるため、物理的・電気的・熱的に分離して配置する。詳細は別文書「車両アーキテクチャ設計書」で扱う。

# 5. 駆動系要件

## 5.1 アーキテクチャ要求

駆動系は、Fail-Safe を許容する。すなわち、片系喪失時には駆動を遮断し、惰性走行 + 制動でMRCを達成する。

駆動遮断の冗長性は、Dual-channel駆動遮断（モータ制御ECU + 電源遮断ECU）で実現する。

## 5.2 性能要求（暫定値）

| 項目 | 要求値（暫定） |
| --- | --- |
| 駆動遮断応答時間 | ≤ 50ms |
| 駆動指令応答（通常時） | ≤ 30ms |
| 回生制動最大減速度 | ≥ 2.0 m/s²（補助的） |

## 5.3 駆動指令断の扱い

駆動指令断（ADSからの駆動指令が一定時間途絶）の場合、車両側は自律的に駆動を遮断し、ADSへ通知する。直近指令の保持時間は ≤ 200ms とする。

# 6. 電源冗長要件

## 6.1 アーキテクチャ要求

12V低電圧（LV）電源系は、LV1とLV2の二系統独立構成とする。HV系（駆動用高電圧バッテリ）からLV1/LV2への電力供給は、独立したDC-DCコンバータで行う。

LV1とLV2は、負荷振り分けにより、片系喪失時にもSafety-Critical機能を維持する。

## 6.2 LV1/LV2の負荷振り分け（暫定）

| LV1主負荷 | LV2主負荷 |
| --- | --- |
| 通常走行ADS（通常走行用ECU）| MRC系ADS（MRC用ECU）|
| プライマリ制動ECU | バックアップ制動ECU |
| EPSコイル系統A | EPSコイル系統B |
| ゲートウェイA | ゲートウェイB |
| カメラ系統A | カメラ系統B |
| プライマリ通信 | バックアップ通信 |
| 灯火（前方） | 灯火（後方）+ ハザード |

## 6.3 補助バッテリ容量

LV補助バッテリは、HV系喪失時にも以下の動作を完遂できる容量を持つこと。

MRC完了までの制御電力（最大3分間）／停止保持（EPB起動 + ハザード継続、最大30分）／ログ書出し（最大2分）／外向けHMI継続（最大10分）／遠隔監視通信（最大10分）。

具体的容量計算は、車両構成確定後に詳細設計で行う。

## 6.4 性能要求（暫定値）

| 項目 | 要求値（暫定） |
| --- | --- |
| LV1/LV2電圧範囲 | 11.0〜14.5V（通常運用域） |
| 電圧低下時の警告閾値 | 11.5V以下 |
| Safety-Critical機能の動作下限 | 9.0V |
| LV片系切替時間 | ≤ 50ms |
| 補助バッテリの予測残時間通知 | 1秒周期、誤差 ≤ 10% |

## 6.5 故障モードと挙動

| 故障モード | 要求挙動 |
| --- | --- |
| HV系喪失 | LV補助バッテリで稼働継続、MRC遷移、ADSへ通知 |
| LV1喪失 | LV2系統でSafety-Critical機能維持、MRC遷移、ADSへ通知 |
| LV2喪失 | LV1系統でSafety-Critical機能維持、MRC遷移、ADSへ通知 |
| LV両系喪失 | 制動・ハザードを物理的に確保、駆動遮断、機械的フェイルセーフ |
| DC-DCコンバータ片系故障 | 残存系で動作、ADSへ通知 |

## 6.6 単一故障での同時喪失禁止

LV1とLV2が、単一故障で同時喪失しないこと。具体的には、共通バスバー、共通ヒューズ、共通ハーネスルートを避ける。物理的セパレーション要件として詳細設計で確認する。

# 7. 通信冗長要件

## 7.1 バックボーン要件

バックボーンネットワークは、Automotive Ethernet（最低1G、推奨10G）+ TSN（IEEE 802.1AS / Qbv / CB / Qci）とする。

冗長構成は、Dual ring または Dual star を前提とする。FRER（IEEE 802.1CB）により、Safety-Critical メッセージのフレーム複製・経路冗長を実装する。

## 7.2 帯域要求（暫定）

| 区分 | 帯域 |
| --- | --- |
| ADS↔車両 Safety-Critical | ≥ 100 Mbps（予約帯域） |
| カメラデータ | ≥ 6 Gbps（圧縮前換算）/ ≥ 1 Gbps（圧縮後） |
| ログ・診断 | ≥ 100 Mbps |
| OTA・遠隔監視 | ≥ 50 Mbps |

## 7.3 遅延・ジッタ要求

| 項目 | 要求値（暫定） |
| --- | --- |
| Safety-Critical 最大遅延 | ≤ 5ms（end-to-end） |
| Safety-Critical ジッタ | ≤ 1ms |
| 時刻同期精度（PTP） | ≤ 1μs |

## 7.4 故障モードと挙動

| 故障モード | 要求挙動 |
| --- | --- |
| 片系リング/スター故障 | FRERで冗長経路に自動切替、ADSへ通知 |
| 両系喪失 | 通信断として扱い、MRC遷移、保守的緊急停止 |
| ゲートウェイ片系故障 | 残存ゲートウェイで動作、ADSへ通知 |
| PTP同期断 | Safety-Critical動作の前提失陥として扱い、MRC遷移要因 |

# 8. センサ取付要件

## 8.1 カメラ構成（7+1台）

本車両に搭載するカメラは、通常走行用カメラ群7台とMRC専用カメラ1台、計8台で構成する。

| **カメラ** | **対象ADS** | **役割** | **接続先** |
| --- | --- | --- | --- |
| フロント広角 | 通常走行ADS | 通常走行・前方広視野 | 通常走行用ECU |
| フロントワイド | 通常走行ADS | 通常走行・前方中遠距離 | 通常走行用ECU |
| 左前方 | 通常走行ADS | 通常走行・左前方視野 | 通常走行用ECU |
| 右前方 | 通常走行ADS | 通常走行・右前方視野 | 通常走行用ECU |
| 左後方 | 通常走行ADS | 通常走行・左後方視野 | 通常走行用ECU |
| 右後方 | 通常走行ADS | 通常走行・右後方視野 | 通常走行用ECU |
| 後方 | 通常走行ADS | 通常走行・後方視野 | 通常走行用ECU |
| **MRC専用カメラ** | **MRC系ADS** | **MRC実行（同一車線内停止）** | **MRC用ECU（直結）** |

通常走行用カメラ群7台は、通常走行用ECUに接続される。1台が失陥しても、通常走行ADSは残存6台で安全な停止位置までの走行と停止を実行できる能力を持つ（Turing Safety Claim 10.3参照）。

MRC専用カメラ1台は、MRC用ECUに直結される。通常走行用カメラ群とは物理・電気・伝送系で完全独立とする。

## 8.2 通常走行用カメラ群とMRC専用カメラの独立性要件

通常走行用カメラ群とMRC専用カメラは、以下のレベルで独立性を確保すること。

| **層** | **独立性要件** |
| --- | --- |
| 物理配置 | 通常走行用カメラ群とMRC専用カメラは異なる取付ブラケット・異なる位置（衝突・振動が同時に同程度に伝わらない配置） |
| 電気・電源 | 通常走行用カメラ群はLV1系、MRC専用カメラはLV2系 |
| 伝送経路 | 別ケーブル・別ハーネス・別コネクタ・別ゲートウェイポート |
| ECU | 通常走行用ECU（通常走行用カメラ群） / MRC用ECU（MRC専用カメラ） |
| キャリブレーション | 個別のキャリブレーションプロセス・個別のキャリブレーションパラメタ管理 |

これにより、Turing Safety Claim 9.9のCCFカテゴリ「同一センサ入力」を設計上排除する。

## 8.3 MRC専用カメラの仕様要件

MRC専用カメラはMRC系ADSの唯一の入力として、以下を満たすこと。

| **項目** | **要求** |
| --- | --- |
| 視野 | 前方広角（同一車線内の前方障害物・車線・近距離歩行者の認識に十分なFOV） |
| 解像度 | MRCタスク（衝突回避・車線維持）に十分な解像度 |
| ダイナミックレンジ | 高HDR（逆光・夜間・トンネル進入等での認識劣化を最小化） |
| フレームレート | 通常走行用と整合（時刻同期可能） |
| 故障率 | MRC capability喪失への寄与を抑える高信頼性等級 |
| 取付剛性 | 走行中の振動・温度変化で健全性要件を維持 |

## 8.4 取付剛性

通常走行用カメラ群およびMRC専用カメラの取付剛性は、走行中の振動・温度変化でキャリブレーションがTuring Safety Claim 10.5のカメラ健全性要件を満たし続けるレベルを保つこと。

定期的な再キャリブレーションプロセス、フィールドでの異常検知方法は、運用文書群で扱う。

## 8.5 環境耐性

カメラ、ISP、伝送系は、車両の動作保証温度範囲、振動規格（ISO 16750）、防水・防塵規格（IP6K9K相当）を満たすこと。

特にMRC専用カメラは、フロントガラス越しの環境曝露（雨・霧・夜間・逆光・汚損・凍結）が通常走行用カメラ群と共通である点を踏まえ、ワイパー有効カバー範囲、ヒーター、防汚コーティング等の補助手段を含めて取付位置を確定する。

# 9. 灯火・HMI要件

## 9.1 ハザード

ハザードランプは、ADS指令と独立にハードウェア起動可能なバックアップ経路を持つこと。具体的には、ADS指令に加え、車両ECUが緊急停止状態を独立に判定してハザードを点灯できる。

## 9.2 外向けHMI

外向けHMI（自動運転中表示、MRC中表示、停止中表示、乗降待ち表示、緊急時表示）の表示要件は、別文書「外向けHMI仕様書」で扱う。

外向けHMIは、Turing Safety Claim 11.6に従い、MRC開始時、停止後、復帰時に明示的な表示を行うこと。

## 9.3 内向けHMI

L4運用時、車内には運転者がいないことを前提とする。一方、乗員に対する状態表示、緊急時の指示、コミュニケーション手段は別文書「内向けHMI仕様書」で扱う。

# 10. サイバーセキュリティ要件

## 10.1 UN-R 155準拠

車両プラットフォームはUN-R 155に準拠すること。詳細は別文書Cybersecurity Caseで扱う。

## 10.2 OEM-Turing分担（CSMS）

| 領域 | 主管 |
| --- | --- |
| 車両ECUのセキュア実装 | OEM |
| ADS構成要素のセキュア実装 | Turing |
| 車載ネットワークセキュリティ | 共同 |
| 製造工程のセキュリティ | OEM |
| 運用時SOC・PSIRT | 共同 |
| 鍵管理（HSM、PKI） | 共同（鍵の所在は協定書で確定） |

## 10.3 OTA Update要件

ISO 24089準拠。OTA更新時の認証、整合性検証、ロールバック、通常走行系/MRC系の不整合バージョン状態（rolling update中）の扱いを規定する。

## 10.4 攻撃面と対策の要件

詳細はCybersecurity Caseで扱うが、車両側で対応すべき攻撃面として、以下を含むこと。
診断ポート（OBD-II、UDS）への不正アクセス、車載Wi-Fi/Cellularへの攻撃、CANバスへの不正注入、ECUファームウェアの改ざん、製造工程での悪意混入、サプライチェーン攻撃。

# 11. 機能安全プロセス要件

## 11.1 ISO 26262準拠

車両ECU群はISO 26262第2版に準拠した開発を行うこと。Safety Element out of Context（SEooC）として開発される共通部品は、本書の要求を含むSafety Manualを伴うこと。

## 11.2 エビデンス提供

OEMは、以下のエビデンスをTuringに提供する。
HARA結果（Item Definition、Hazard analysis、Risk assessment）／ASIL分解結果／Safety Goal、Functional Safety Concept、Technical Safety Concept／FMEA、FTA、DFA（Dependent Failure Analysis）／Safety Mechanism実装エビデンス／検証・妥当性確認結果／Safety Case／Safety Manual。

提供フォーマット、トレーサビリティ、PLM・DOORS等のシステム連携は、別文書「エビデンス提供要件書」で確定する。

## 11.3 共通原因故障分析

車両プラットフォーム側のCCF分析結果を、Turing側のADS-CCF分析と統合可能な形で提供する。特に、共通電源、共通通信、共通熱、共通振動、共通水/塵、共通電磁ノイズ、共通サプライチェーン由来欠陥を扱うこと。

# 12. 量産・サプライチェーン要件

## 12.1 量産品質保証

VIN単位での車両構成・ソフトバージョン管理を行うこと。出荷時、メンテナンス後、OTA更新後の構成を、Turingが照会可能な形で管理する。

## 12.3 製造工程

ADS搭載に関わる製造工程（センサ取付・キャリブレーション、ECU取付、ソフトウェア書込、初期動作確認）は、TuringとOEMが共同で確定する。

## 12.4 フィールドメンテナンス

事故・修理後の再キャリブレーション、ソフトウェア更新、構成変更時の安全論証維持プロセスを、別文書「フィールドメンテナンス要件書」で扱う。

# 15. 数値確定プロセス

本書の暫定値を確定値に移行するプロセス。

ステップ1：HARA結果が確定次第、ASIL目標と性能要求の整合を確認。
ステップ2：お台場ODD定義書が確定次第、停止距離・MRC性能要求を確定。
ステップ3：プロトタイプ車両での検証エビデンス取得後、暫定値を実測ベースで見直し。
ステップ4：Phase 1実証データを反映し、Phase 2型式指定取得時に確定。
ステップ5：Phase 2運用後、Phase 3量産時の最終確定。

各ステップでの数値変更は、本書改版または別協定書で管理し、関係するTuring Safety Claim章および衛星文書群に反映する。
