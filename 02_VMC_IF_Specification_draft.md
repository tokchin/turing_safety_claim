# 車両制御I/F仕様書（VMC I/F Specification）骨子ドラフト

本書は、Turing_Safety_Claim.md の衛星文書として位置付ける、車両制御I/F（Vehicle Motion Control Interface, VMC I/F）の仕様骨子である。

本書の目的は、TuringのADS（通常走行ADSと MRC系ADSの異種ECU冗長構成）と、OEMが提供するL4対応車両プラットフォーム（制動、操舵、駆動、灯火、電源、通信ECU群）との間の制御I/Fを、共同開発の起点として早期に確定することである。

本書は契約書ではない。本書は、OEMとサプライヤを巻き込んだ詳細仕様化、および型式指定取得に向けた要求仕様化の起点として用いる。

---

# 1. 本書の位置付け

## 1.1 上位文書との関係

本書は、Turing_Safety_Claim 第2版の第10章（センサ・計算機・車両制御系の失陥安全）、第17章（責任分界・技術）、Appendix I（責任分界マトリクス）の参照先文書である。

本書は、車両要求仕様書（03_Vehicle_Requirements_Specification_draft.md）の一部を構成し、特に通信プロトコル・指令フォーマット・Safety Mechanismを詳細化する位置にある。

## 1.2 本書の確定スコープ

本書では、ADSと車両プラットフォームの間の論理I/Fを確定する。具体的には、指令系（ADS → 車両）、フィードバック系（車両 → ADS）、Arbitration（通常走行系/MRC系の入力裁定）、Safety Mechanism（E2E protection、Plausibility check、健全性監視）を扱う。

## 1.3 本書の確定外スコープ

物理層の選定（Automotive Ethernet / CAN-FD / FlexRay の混在比率）、コネクタ形状、ピンアサイン、ECU内部実装、サプライヤ固有のソフトウェアスタックは、本書では確定しない。これらは詳細仕様化フェーズでサプライヤを含めて確定する。

# 2. システム境界とI/F定義の前提

## 2.1 境界の定義

ADS側（Turing主管）：通常走行用ECU、MRC用ECU、安全監視系ECU、ADSと車両ECUを接続するゲートウェイのADS側ポート。

車両側（OEM主管）：制動ECU、操舵ECU、駆動ECU（モータ制御または変速機制御を含む）、ボディECU（灯火、ハザード、ホーン、ドアロック、外向けHMI）、電源管理ECU、ゲートウェイの車両側ポート、車両状態センサ。

I/F：上記の境界をまたぐ全ての論理メッセージ、信号、診断データ、時刻同期、健全性ハートビート、Safety Mechanism信号を含む。

## 2.2 前提とするバックボーン

バックボーンネットワークはAutomotive Ethernet（1G/10G）+ TSN を前提とする。Time-Sensitive Networking（IEEE 802.1AS / Qbv / CB / Qci）により、Safety-Critical Pathの帯域・遅延・冗長を保証する。

CAN-FDは、レガシーECUとの接続および診断・低速制御信号に限定して併用する。

## 2.3 冗長構成

I/F全体としてDual-channel冗長を前提とする。具体的には、ADS側は通常走行用ECUとMRC用ECUが独立にゲートウェイへ接続し、車両側ECU群はDual ringまたはDual starでゲートウェイから接続される。

冗長構成の詳細は、車両要求仕様書（03_Vehicle_Requirements_Specification_draft.md）の「通信冗長要件」を参照。

# 3. 指令系I/F（ADS → 車両）

## 3.1 指令の基本構造

ADSから車両への指令は、以下のカテゴリで構成する。

| カテゴリ | 主な信号 | 周期 | ASIL目標 |
| --- | --- | --- | --- |
| 操舵指令 | 目標舵角 / 目標舵角速度 / トルク制限 | 10〜20ms | ASIL-D |
| 加減速指令 | 目標加速度 / 目標減速度 / 駆動・制動配分要求 | 10〜20ms | ASIL-D |
| 制動指令 | 目標減速度 / 緊急ブレーキ要求 / 駐車ブレーキ要求 | 10ms | ASIL-D |
| 駆動指令 | 駆動許可 / 駆動遮断要求 / 出力制限 | 50ms | ASIL-C |
| 灯火・HMI指令 | ハザード / ウインカ / ヘッドライト / ホーン / 外向けHMI | 100ms | ASIL-B |
| モード指令 | 通常走行 / MRC / 停止保持 / 復帰禁止 | イベント | ASIL-D |
| ドア・ロック指令 | ドア施錠 / 解錠 / 乗降許可 | イベント | ASIL-B |

## 3.2 操舵指令の詳細

目標舵角は、車両座標系での前輪等価舵角（degree）とする。範囲・分解能・更新レートは詳細仕様化フェーズで確定する。

目標舵角速度は、ADSが車両に許容する最大舵角変化レート（degree/sec）を示す。これによりADSが車両側のActuator動的制限を意識した指令を出すことを保証する。

トルク制限は、操舵Actuatorの出力トルク上限要求（Nm）を示す。ドライバ介入や物理障害（縁石への乗り上げ）でActuatorが過大トルクを発生しないための上限を、ADSが指令する。

## 3.3 加減速指令の詳細

目標加速度（および目標減速度）は、車両前後方向の目標加速度（m/s²）とする。正負を符号で扱う。

駆動・制動配分要求は、加減速の実現手段として駆動と制動のどちらを主とするかをADSが推奨するパラメタである。回生ブレーキとの整合、エネルギー効率、制動Actuatorの摩耗管理に用いる。最終的な配分判断は車両側で行う。

緊急ブレーキ要求は、通常の減速指令と独立した別チャネルで送信する。緊急ブレーキ要求が立った場合、車両は最大減速度で減速する。

## 3.4 モード指令と状態遷移

ADSは、車両に対してモード指令を発行する。車両は、現在の状態と整合する場合のみモード遷移を実行し、整合しない場合は遷移要求を拒否する旨を返す。

| モード指令 | 意味 | 遷移可能元 |
| --- | --- | --- |
| 通常走行 | 通常走行モード | 起動完了 |
| MRC開始 | MRCモードへ遷移 | 通常走行 |
| 停止保持 | MRC完了後の停止保持 | MRCモード |
| 復帰禁止 | 通常走行への自動復帰を禁止 | 全モード |
| シャットダウン | 安全シャットダウン | 停止保持 |

通常走行モードへの自動復帰は、本書のSafety Mechanism（後述）でブロックする。

## 3.5 指令フレーム構造

各指令メッセージは、以下のフィールドを含む。

ヘッダ：送信元ID（通常走行系/MRC系）、メッセージID、シーケンス番号、タイムスタンプ（IEEE 1588 PTP）。

ペイロード：上記カテゴリごとの目標値・制限値・モード。

セキュリティ・健全性：CRC32、E2E protection（AUTOSAR E2E Profile相当のCounter + Checksum）、メッセージ認証コード（CMAC、UN-R 155準拠CS処理）。

優先度：Safety-Critical / Time-Critical / Best-Effort の三段階。TSNのCBS/Qbvで分離する。

# 4. フィードバック系I/F（車両 → ADS）

## 4.1 フィードバックの基本構造

車両からADSへのフィードバックは、以下のカテゴリで構成する。

| カテゴリ | 主な信号 | 周期 |
| --- | --- | --- |
| 車両運動状態 | 車速 / 操舵角実値 / ヨーレート / 縦加速度 / 横加速度 | 10ms |
| Actuator応答 | 制動圧 / 駆動トルク実値 / 操舵トルク実値 / 各輪回転速度 | 10ms |
| ECU健全性 | 各ECUのハートビート / 自己診断結果 / DTC | 100ms |
| 残存制御能力 | 制動残存能力 / 操舵残存能力 / 駆動残存能力 / 電源残存能力 | 100ms |
| 環境状態 | 車両姿勢 / 路面摩擦推定 / 車両周囲の通信状態 | 200ms |
| 故障状態 | アクティブ故障コード / 故障重大度 / 縮退モード状態 | イベント |

## 4.2 残存制御能力の通知

これは多重系ADSの安全論証で特に重要な信号である。車両側ECUは、各Actuatorが現時点で発揮可能な性能を、ADSに対して定常的に通知する。

| 信号 | 意味 |
| --- | --- |
| 制動残存能力 | 現時点で発揮可能な最大減速度（m/s²） |
| 操舵残存能力 | 現時点で発揮可能な最大舵角速度・最大トルク |
| 駆動残存能力 | 現時点で発揮可能な駆動力範囲 |
| 電源残存能力 | LV1系・LV2系の電圧・残容量・予測持続時間 |

ADSは、この情報を基にMRC開始判断、MRC位置選定、MRC挙動の調整を行う。

## 4.3 ECU健全性ハートビート

各ECUは固定周期でハートビートを送信する。ハートビート断、シーケンスずれ、CRC異常を、ADSの安全監視系および車両ゲートウェイの双方が独立に検知する。

ハートビート断のFTTI（Fault Tolerant Time Interval）目標は、車両要求仕様書で確定する。

# 5. Arbitration（通常走行系/MRC系の入力裁定）

## 5.1 Arbitrationの目的

通常走行ADSとMRC系ADSの双方が、車両ゲートウェイに対して指令を送信しうる。Arbitrationの目的は、二重指令、無指令、矛盾指令、なりすまし指令を防止することである。

## 5.2 Arbitrationの基本ルール

通常走行時は、通常走行ADSの指令を採用し、MRC系ADSの指令はリスニングモード（送信抑制 or 監視のみ）とする。

フェイルオーバー要求時は、通常走行ADSの指令を遮断し、MRC系ADSの指令を採用する。フェイルオーバーは、安全監視系が通常走行系の異常を検知した場合、または通常走行系自身が自己異常を申告した場合に発火する。

両系とも異常時は、両系統の指令を遮断し、車両側で保守的な緊急停止挙動（最大減速 + ハザード + 駆動遮断 + ステアリング保持）を実行する。

## 5.3 Arbitration実装の所在

Arbitration機能は、車両側ゲートウェイに実装する。これにより、通常走行系/MRC系の双方の故障に対して独立な裁定を行える。

ゲートウェイ自体の故障に備え、Arbitration機能は二重化する。

## 5.4 Arbitration信号

ADSは、自系統の状態をArbitration要求として明示する。

| 信号 | 意味 |
| --- | --- |
| 通常走行系健全 | 通常走行系が指令送信を継続する |
| 通常走行系自己異常申告 | 通常走行系が異常を自己検知し、フェイルオーバーを要求する |
| MRC系健全 | MRC系が指令送信を引き継ぐ準備完了 |
| MRC系MRC実行中 | MRC系がMRCモードで指令を送信中 |
| 両系異常 | 両系統が異常状態 |

安全監視系は、これらの申告を独立に検証し、矛盾がある場合は車両側で保守的緊急停止に移行する。

# 6. Safety Mechanism

## 6.1 E2E Protection

全てのSafety-Critical メッセージにE2E Protection（Counter + Checksum、AUTOSAR E2E Profile 4または5相当）を適用する。受信側は、Counter連続性、Checksum妥当性、Timeout、Sequence reorderingを検出する。

異常検出時の動作は、メッセージ種別ごとに定義する。指令系の異常は即時にArbitrationを発火させ、フィードバック系の異常は健全性低下として扱う。

## 6.2 Plausibility Check

ADS指令と車両物理応答の整合を、車両側で独立に検証する。例えば、操舵指令と実舵角の乖離、加速度指令と縦加速度の乖離、駆動指令と駆動トルクの乖離を、Plausibility Checkとして監視する。

本Plausibility Checkは、指令の物理的実現状態（指令されたとおりに車両が応答したか）を確認するものであり、指令そのものの妥当性（その指令が状況的に適切か）を判定するものではない。指令妥当性のうち、「急ブレーキを出すべきか否か」など外部認識を要する判定は、E2E AI自身の出力品質に委ねる（Turing Safety Claim 8.6）。

Plausibility失敗時は、車両側がArbitrationに対して異常申告を発出する。

## 6.3 指令範囲制限

ADS指令は、車両側で物理的・運用的範囲に制限される。指令範囲を超えた値は、車両側でクリップした上で、Plausibility失敗として扱う。

## 6.4 Cybersecurity Mechanism

全てのSafety-Critical メッセージにメッセージ認証（CMAC）を適用する。鍵管理、鍵更新、鍵失効プロセスは、Cybersecurity Caseで詳細化する（UN-R 155準拠）。

# 7. 時刻同期

## 7.1 時刻同期方式

IEEE 1588 PTP（Precision Time Protocol）をバックボーンEthernet上で運用する。Grandmaster Clockの選定（GNSS同期 or ADS側Master）、Boundary Clockの配置、同期精度目標は詳細仕様化フェーズで確定する。

## 7.2 時刻同期失陥時の扱い

PTP同期断、Grandmaster切替、同期精度劣化を、ADS安全監視系および車両ゲートウェイの双方が独立に検知する。

時刻同期失陥は、Safety-Critical制御の前提条件失陥として扱い、MRC遷移要因に含める。

# 8. 起動・シャットダウン・モード遷移

## 8.1 起動シーケンス

電源投入から通常走行モード許可までのシーケンスを定義する。具体的には、各ECUの自己診断完了、ハートビート確立、時刻同期確立、Cybersecurity鍵確立、ADS自己診断完了、通常走行系・MRC系の双方が健全であることの確認、を順に実施する。

## 8.2 シャットダウンシーケンス

通常シャットダウン、緊急シャットダウン、停止保持→シャットダウンの3つのシーケンスを定義する。緊急シャットダウンでは、ログ書出し完了、停止状態の保持、ハザード継続、駆動完全遮断、を保証する。

## 8.3 モード遷移マトリクス

通常走行 ↔ MRC開始 ↔ 停止保持 ↔ シャットダウン の遷移可能性、遷移トリガ、遷移後の保証動作を表形式で定義する。

通常走行への自動復帰は、本書のSafety Mechanismで一律ブロックする。手動復帰は、地上側の許可プロセス（遠隔監視オペレータの明示操作）を経て初めて可能とする。

# 9. 診断・ログ

## 9.1 DTC（Diagnostic Trouble Code）

UDS（ISO 14229）に準拠したDTCを定義する。DTC体系は、ADS側DTC、車両側DTC、共同DTC（境界故障）の3グループで管理する。

## 9.2 DSSAD相当ログ項目

UN-R 157のDSSAD要件に対応するため、I/Fをまたぐ全ての指令、フィードバック、Arbitration事象、モード遷移、Safety Mechanism発火、Cybersecurity事象を、ログとして記録する。

ログ項目の詳細は、Turing Safety Claim 第2版第13章および別文書「ログ項目定義書」で確定する。

## 9.3 ログのDual recording

ログは、ADS側と車両側で独立に記録する（Dual recording）。一方が消失または改ざんされた場合に、他方から再現可能とする。

# 10. 性能要求

## 10.1 遅延要求

| 区間 | 目標値（暫定） |
| --- | --- |
| ADS指令送信→車両Actuator動作開始 | < 30ms |
| 車両状態取得→ADS入力到達 | < 20ms |
| 緊急ブレーキ要求→制動開始 | < 50ms |
| ハートビート断検知→Arbitration発火 | < 100ms |
| Arbitration発火→MRC系指令採用開始 | < 50ms |
| Cascading（通常走行系異常→MRC系MRC開始） | < 200ms（FTTI目標） |

具体値は、車両要求仕様書および詳細試験で確定する。

## 10.2 帯域要求

Safety-Critical Pathに必要な帯域を予約する。Best-Effortトラフィックとは独立に、TSNのCBS/Qbvで保護する。

## 10.3 可用性要求

I/F全体の可用性目標は、車両要求仕様書で確定する。Dual-channel冗長と、片系喪失時のMRC継続能力を含む。

# 11. 検証・受入

## 11.1 検証項目

I/F検証は、以下の観点で実施する。
信号レベル（フォーマット、CRC、E2E）／プロトコルレベル（TSN、PTP、UDS）／機能レベル（指令-応答、Arbitration、モード遷移）／Safety Mechanism（故障注入、Plausibility失敗）／Cybersecurity（鍵管理、認証、攻撃シナリオ）。

## 11.2 故障注入試験

各I/F信号に対して、以下の故障注入を実施する。
信号断、信号遅延、信号化け、シーケンスずれ、CRC破壊、E2E Counter停止、メッセージ認証失敗、矛盾指令、なりすまし、Replay攻撃。

## 11.3 受入基準

各検証項目について、合格基準、判定方法、エビデンス保存形式を確定する。詳細は別文書「VMC I/F検証計画書」で扱う。

# 12. 責任分界

## 12.1 I/F両端の責任

ADS側I/F実装の責任：Turing主管。
車両側I/F実装の責任：OEM主管。
I/F仕様の確定責任：共同（本書をベースとする）。

## 12.2 境界故障の扱い

I/Fをまたぐ故障（例：CRC失敗、E2E Counter不整合）は、いずれの側に原因があるかを切り分けるために、双方独立にログを保存する。原因究明プロセスは、Turing Safety Claim第18章「事故・インシデント対応プロセス」に従う。

## 12.3 仕様変更管理

本書の改版は、Turing-OEM共同で承認する。改版時は、後方互換性、移行期間、車両構成・ソフトウェアバージョン管理（PLM・DOORS等）への反映を確認する。
