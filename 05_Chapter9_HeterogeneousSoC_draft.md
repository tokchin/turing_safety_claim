## 9.3 MRC系ADSの役割（**改訂**）

MRC系ADSは、通常走行ADSに失陥が生じた場合に、自動的に起動または制御権限を引き継ぎ、MRCモードへ遷移する。

MRC系ADSは、通常走行を完全に代替することを主目的としない。一方で、MRC系ADSの役割は、能力の制限ではなく行動範囲の制限として整理する。

すなわち、MRC系ADSはODD内のあらゆる動的環境（歩行者、自転車、他車両、信号、横断歩道、交差点、緊急車両等）に対する衝突回避能力を持つ。一方で、MRC系ADSの行動目的はMRC達成に限定し、車線変更、追越し、合流、後退、不確実な交差点進入等は行わない。

MRC系ADSの目的は、以下である。

1. 通常走行ADSの失陥時に無指令状態を避ける

2. 危険な通常走行ADS出力を遮断する

3. ODD内の動的環境に対する衝突回避を継続する

4. MRCモードへ遷移する

5. 車両を減速または安全位置に停止させる

6. MRC完了後に再発進を制限する

MRC系ADSがこれらを成立させるための実装は、9.4に示す異種ECU冗長アーキテクチャを採用する。

## 9.4 異種ECU冗長アーキテクチャ（**新設**）

### 9.4.1 異種ECU冗長を採用する理由

Turingは、通常走行ADSとMRC系ADSのハードウェア実装として、異なるECUを採用する。具体的には、通常走行ADSには通常走行用ECU、MRC系ADSにはMRC用ECUを用い、両ECUは異なる世代・異なる物理設計のシリコンで構成する。

本書では実装ECUの具体名を扱わず、機能名「通常走行用ECU」「MRC用ECU」のみを用いる。具体的なECU実装、世代、サプライヤ、Safety Report等は車両要求仕様書、AI Safety Case、Cybersecurity Caseで管理する。

この構成を採用する理由は、以下である。

通常走行の判断は、高度な状況理解、長期的計画、複雑な交通参加者の予測を要するため、高性能AI処理に対応した通常走行用ECUが必要である。

MRC遂行は「ぶつからずに止まる」「同一車線内に安全停止する」というタスクに限定される。MRC専用E2E AIが扱う範囲はODD内の衝突回避と安全停止であり、MRCタスクに十分な性能を持つMRC用ECUで実装する。

通常走行ADSとMRC系ADSを異なるECUで構成することにより、シリコン、ファームウェア、OS、AIモデル、ツールチェーン、検証フレームワークの全レベルで自然なDiversityが達成される。これにより、共通原因故障（CCF）の主要なリスクが設計上排除される。

両ECUを同一サプライヤから調達することにより、開発ツールチェーン、Safety Manual、認証ステータス、長期供給性に関する管理コストを抑える。これは、独立したサプライヤ間のDiversityを採る場合に比べ、性能・コスト・量産性のバランスを取る現実的な構成である。

### 9.4.2 異種ECU冗長の構成概要

| 項目 | 通常走行ADS | MRC系ADS |
| --- | --- | --- |
| 主目的 | 通常走行 | MRC達成（同一車線内安全停止） |
| AIモデル | 通常走行用E2E AI | MRC専用E2E AI |
| 計算機 | 通常走行用ECU（高性能AI処理向け） | MRC用ECU（MRCタスク十分性能） |
| シリコン世代 | 高性能世代 | 異世代（前世代相当） |
| AIモデル開発系統 | 通常走行モデル開発チーム | MRC専用モデル開発チーム |
| 学習データ | 通常走行データセット（広汎な走行場面） | MRC専用データセット（衝突回避・車線内停止シナリオ重視） |
| 入力カメラ | 7台（フロント広角、フロントワイド、左右前方、左右後方、後方） | MRC専用カメラ1台（前方広角、MRC用ECU直結） |
| 出力 | 舵角・加速度・モード指令 | 舵角・加速度・MRC指令 |
| 電源 | LV1主給電 | LV2主給電（独立） |
| 通信ゲートウェイ | プライマリ | バックアップ |

### 9.4.3 異種性のレベル

異種ECU冗長の「異種性」は、以下のレベルで成立する。

| レベル | 異種性の内容 |
| --- | --- |
| シリコン | 通常走行用ECUとMRC用ECUは異なる世代、異なる物理設計、異なる製造マスク、異なる製造ロット |
| ファームウェア | 通常走行用ECUとMRC用ECUは個別のSDKバージョン、個別のブートチェーン |
| OS | 個別最適化された個別カーネル |
| ミドルウェア | ADS基盤の独立実装 |
| AIモデル | 異なるアーキテクチャ、異なる学習データ、異なる学習ハイパパラメタ、異なる検証セット |
| ツールチェーン | 個別の学習・検証・デプロイパイプライン |
| 開発チーム | 通常走行チームとMRC専用チームの組織分離（systematic failure共通化を防ぐ） |

### 9.4.4 共通サプライヤとしての残存リスクと対策

両ECUを同一サプライヤから調達することに起因する残存CCFリスクは、以下である。

サプライチェーン由来の共通脆弱性（同一サプライヤの世代横断的な物理設計欠陥、共通ファブリック由来の脆弱性）／同一サプライヤSDKの共通バグ／共通サプライヤの長期供給性リスク／共通サプライヤのCS脆弱性公開時の同時被災／製造工程のソフトウェア書込ツールの共通性。

これらの残存リスクへの対策は、以下を含む。

ECUサプライヤSafety Manualの双方完全適用と、通常走行用ECUとMRC用ECU個別のSafety Reportの取得／ECUサプライヤPSIRTからの脆弱性情報を運用後監視で継続フォロー／長期供給契約と代替サプライヤの探索（Phase 3で本格化）／OEM-Turing-ECUサプライヤ三者による定期的なSafety/Cybersecurityレビュー／ソフトウェア書込ツールのCSハードニングと製造工程CS（UN-R 155準拠）。

これらの残存リスクは、本書第21章の残存リスクとして明示する。

### 9.4.5 異種ECU冗長の安全主張

異種ECU冗長アーキテクチャの安全主張は、以下である。

*通常走行ADSが失陥した場合でも、MRC系ADSは異なるシリコン、異なるファームウェア、異なるOS、異なるAIモデルにより独立に動作し、ODD内の動的環境への衝突回避能力を保ったまま、MRCモードへ遷移し、車両単独でリスク最小状態に到達する。*

この安全主張は、本書第4章の最上位安全主張を、ハードウェア・ソフトウェアの両レベルで支える。

### 9.4.6 異種ECU冗長によるCCF設計排除

本構成により、以下のCCFは設計上排除される。

| CCFカテゴリ | 設計排除の根拠 |
| --- | --- |
| 同一シリコンの物理欠陥 | 通常走行用ECUとMRC用ECUは異なる物理設計、異なる製造マスク、異なる製造ロット |
| 同一ファームウェアの欠陥 | 個別ファームウェアスタック |
| 同一OSの欠陥 | 個別カーネル・個別BSP |
| 同一AIモデルの欠陥 | 異なるアーキテクチャ、異なる学習データ |
| 同一学習バイアス | 異なる学習データセット、異なる検証セット |
| 同一systematic failure | 別チームによる独立開発 |
| 同一ツールチェーン由来欠陥 | 個別パイプライン |
| 同一センサ入力の劣化 | 通常走行用カメラ群7台とMRC専用カメラ1台を物理・電気・伝送系で完全独立に配置（10章、車両要求仕様書第8章） |

設計排除されないCCF（残存CCF）は、9.4.4および9.10で扱う。

### 9.4.7 性能・コスト・量産性のバランス

完全に独立したサプライヤ間のDiversity（例：通常走行用ECU + 別サプライヤのサーバ級ECU）と比較し、本構成は以下のメリットを持つ。

開発ツールチェーン・Safety Manual・認証手続きの管理コストが片方分で済む／両ECUの長期供給を同一サプライヤで一括管理できる／OEM-ECUサプライヤ-Turing三者の協業体制を統一できる／量産時の検証・品質保証プロセスを統一できる。

一方、Phase 3以降の量産規模拡大時には、共通サプライヤ依存リスクの定期評価と、必要に応じた代替サプライヤ検討を継続する。

## 9.10 共通原因故障の考え方（**全面拡充**）

### 9.10.1 多重系ADSにおけるCCFの定義

多重系ADSでは、単に系統を二重化するだけでは不十分である。本書では、共通原因故障（Common Cause Failure, CCF）と従属故障（Dependent Failure）を、以下のように整理する。

| 概念 | 定義 |
| --- | --- |
| 共通原因故障（CCF）| 同一の根本原因により、通常走行ADSとMRC系ADSが同時または近接時刻で機能喪失する故障 |
| Cascading Failure | 一系統の故障が、I/F、信号、物理的近接、共通リソースを介して、他系統へ波及する故障 |
| Latent Fault | 直ちに機能喪失を引き起こさないが、残存し、別の故障と組み合わさって安全機能を喪失させる故障 |

### 9.10.2 CCFカテゴリと本書の対応

CCFを設計上の対応によって以下に分類する。

| CCFカテゴリ | 例 | 設計上の対応 |
| --- | --- | --- |
| 同一シリコン由来 | 同型ECUの物理欠陥 | 9.4 異種ECU冗長で**設計排除** |
| 同一ファームウェア由来 | 同一FWのバグ | 9.4 異種ECU冗長で**設計排除** |
| 同一AIモデル由来 | 同一モデルの予測バイアス | 9.4 異種ECU冗長で**設計排除** |
| 同一systematic failure | 同一開発チームの設計過誤 | 9.4 異種ECU冗長（チーム分離）で**設計排除** |
| 共通電源 | LV1喪失で両系停止 | 車両要求仕様書第6章 LV1/LV2分離で**設計対策** |
| 共通通信経路 | バックボーン1系統喪失 | 車両要求仕様書第7章 Dual ring + FRERで**設計対策** |
| 共通時刻同期 | PTP Grandmaster失陥 | VMC I/F仕様書第7章 PTP冗長で**設計対策** |
| 共通環境（熱・振動・水・電磁ノイズ）| 同一筐体内の同時被災 | 物理的セパレーションで**設計対策** |
| 共通センサ入力劣化 | カメラ全系の汚損 | 車両要求仕様書第8章 + 健全性監視で**設計対策** |
| 共通サプライヤ脆弱性 | NVIDIA共通CVE | 9.4.4の運用後監視で**運用対策** |
| 共通製造ロット欠陥 | 同一ロット由来不良 | 製造ロット分散発注で**運用対策** |
| 共通OTA更新不具合 | 不正な同時更新 | rolling update + ロールバックで**運用対策** |

### 9.10.3 Cascading Failureの考え方

Cascading Failureは、一系統の故障が他系統へ波及する故障である。多重系ADSでは、以下のパスが想定される。

通常走行ADSの異常出力が、車両制御I/Fを介して、車両物理応答に影響し、その車両応答がMRC系ADSの判断を歪める／通常走行ADSの異常がArbitration信号を歪め、Arbitration自体を誤動作させる／通常走行ADS筐体の発熱・振動・電磁ノイズが、MRC系ADS筐体に物理的に波及する。

これらに対する対策は、Arbitrationの車両側独立実装（VMC I/F仕様書第5章）、車両側Plausibility Check（VMC I/F仕様書第6.2章）、物理的セパレーション（車両要求仕様書）、安全監視系の独立性確保（9.5）で扱う。

### 9.10.4 Latent Faultと診断カバレッジ

Latent Fault（潜在故障）は、直ちには機能喪失を起こさないが、別の故障と組み合わさったときに安全機能を喪失させる故障である。

本書では、以下のレベルで診断カバレッジを設計する。

通常走行時の通常走行ADSに対する安全監視（9.5）／MRC系ADSの定期的な健全性自己診断（起動時 + 走行中の周期的セルフテスト）／車両ECU群のハートビートと自己診断（VMC I/F仕様書第4.3章）／HW Built-In Self-Test（BIST）と起動時自己診断／センサ健全性の継続監視（10.4）。

ISO 26262 Part 5のSPFM（Single Point Fault Metric）、LFM（Latent Fault Metric）、PMHF（Probabilistic Metric for Hardware Failures）の定量目標は、車両要求仕様書および各ECUのSafety Manualで確定する。

### 9.10.5 残存CCFの管理

設計上排除されない残存CCFは、以下のプロセスで管理する。

DFA文書での明示的列挙／Phase 1〜3を通じた運用後監視／NVIDIA PSIRTからの脆弱性情報フォロー／OEM-Turing-NVIDIA三者の定期的Safety/CSレビュー／重大事象時のリコール対応プロセス（第18章）／本書第21章での残存リスクとしての明示。

### 9.10.6 CCF対応の安全主張

異種ECU冗長と上記のCCF対策により、本書のCCF対応の安全主張は以下である。

*通常走行系とMRC系の同時機能喪失は、シリコン・ファームウェア・OS・AIモデル・systematic failureに起因しては設計上発生しない。共通電源・通信・環境・センサ入力・サプライヤ・ロット・OTAに起因するCCFについては、車両アーキテクチャの分離設計と運用後監視により、許容残存リスク水準まで低減する。*

## 9.11 MRC系ADS実装方針（**書き換え**）

初版9.10では、パターンA（同一構成MRC）とパターンB（縮退構成MRC）の2案を並列検討としていた。本第2版では、9.4に示す**異種ECU冗長アーキテクチャ**を採用方針として確定する。

これは初版のパターンB系列に対応する構成であるが、初版B案の「縮退構成」「軽量モデル」という表現は、能力の限定を強調しすぎる嫌いがあるため、本第2版では「異種ECU冗長による行動範囲限定MRC」と整理し直す。

採用方針確定の根拠は、以下である。

CCF耐性の観点では、異種ECU構成によりHWレベルのCCFリスクが設計上排除される（9.4.6）／MRCモードでの安全側挙動の観点では、MRC用ECU上で動作するMRC専用E2E AIがMRC専用カメラを入力として同一車線内の衝突回避と安全停止を実行できる（9.3）／検証性の観点では、MRC専用モデルの検証範囲が限定的であり、衝突回避シナリオに集中できる／コスト・量産性の観点では、両ECUが同一サプライヤであることで管理コストを抑えられる（9.4.7）。

具体的なECU個別設計、I/F、Arbitration実装、MRC専用E2E AI、検証手順、量産プロセスは、衛星文書群（9.4.8参照）で具体化する。
