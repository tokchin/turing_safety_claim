# 0. 要約

## 0.1 本書の目的

本書は、Turingが開発するL4自動運転車両について、安全性をどのような考え方で成立させ、どのような構造で説明するかを整理する安全論証文書である。

本書では、L4自動運転車両の安全性を、単一の性能指標や単一の試験結果で説明しない。Turingの安全論証では、通常走行時のE2E AIによる安全性と、失陥時の多重系ADSによるMRC達成性を分けて整理する。

TuringのL4自動運転車両は、通常走行時には、複数のカメラ画像（通常走行用カメラ群7台）を主入力とし、舵角指示および加速度指示を出力するE2E AIを中核として走行する。一方で、ADS、センサ、計算機、通信、電源、制動、操舵等に失陥が生じた場合には、通常走行を無理に継続するのではなく、**通常走行用ECUとMRC用ECUからなる異種ECU冗長**による多重系ADS、安全監視系、MRC専用カメラ1台、車両制御I/FによりMRCモードへ遷移し、車両単独でリスク最小状態に到達することを目指す。

本書（第2版）は、TuringのL4安全論証として、安全主張、システム前提、E2E AIの位置づけ、多重系ADSの位置づけ（異種ECU冗長を中核とする）、MRCの基本思想（MRC専用カメラ1台での同一車線内安全停止）、遠隔監視の位置づけ（特定自動運行制度との接続を含む）、ログ・記録保全（UN-R 157 DSSAD要件との対応）、ISO規格・UN規則・国内法令の使い方（対応の四区分）、サイバーセキュリティ・法令制度・型式指定との接続、段階的社会実装（Phase 1〜3）、残存リスクの考え方を示す。

具体的な試験条件、合格基準、検証手順、定量的な受入基準、FMEA/FTA等の詳細は、本書では確定しない。これらは、本書を最上位とする衛星文書群（AI Safety Case、ODD定義書、HARA、車両要求仕様書、VMC I/F仕様書、Cybersecurity Case、モデル更新管理書、ログ項目定義書、運用文書群、特定自動運行計画書等）で具体化する。

## 0.2 TuringのL4安全主張

TuringのL4安全論証における最上位安全主張は、以下である。

*TuringのL4自動運転車両は、定義されたODD内において、車両起因の重大事故および重大インシデントの発生率が、別途定める許容しきい値を下回ることを主張する。本主張は、(a) 通常走行リスクが許容範囲内であること、(b) 失陥時にMRCが成立しリスク最小状態に到達すること、の2つのサブゴールにより支える。サブゴール(a)(b)の信頼性は、運用後監視、更新管理、共通原因故障（CCF）・Cascading Failure・Latent Faultの継続評価等の支援的論証により維持する。*

許容しきい値そのものの数値は本書では確定せず、しきい値の定義方針のみを示す。定義方針は、同一ODDにおける人間運転者の事故率等を比較ベンチマークとする**Positive Risk Balance（PRB）型**を基本とする。具体的な数値、合格基準、検証手順は後続の証拠文書で具体化する。

サブゴール(a)(b)を成立させる基本方針は、次のとおりである。

| **領域** | **基本方針** |
| --- | --- |
| 通常走行安全 | 通常走行用カメラ群7台を入力とし舵角・加速度を出力するE2E AIの閉ループ車両挙動として論証する |
| 失陥時安全 | E2E AIの通常機能で故障を吸収せず、通常走行用ECUとMRC用ECUの異種ECU冗長による多重系ADS、安全監視、MRCで扱う |
| MRC | 故障時の目的を通常走行継続ではなく、MRC専用カメラ1台を用いた同一車線内の安全停止とする |

| 信頼性維持 | 運用後監視、更新管理、CCF・Cascading Failure・Latent Faultの継続評価により安全主張を補強する |

主要HazardおよびSafety Goalの分解、許容しきい値の数値定義は、衛星文書HARAおよび後続の証拠文書で具体化する。

# 1. 本書の位置づけ

## 1.1 背景

L4自動運転車両では、定義された走行環境条件、すなわちODD内において、運転者が運転操作を引き継ぐことを前提にしない。そのため、通常走行時の性能だけでなく、故障、機能低下、ODD逸脱、通信断、遠隔監視断などの状況においても、車両自身がリスクを下げる必要がある。

TuringのL4自動運転システムは、通常走行の中核として、カメラ画像から舵角指示および加速度指示を出力するE2E AIを用いる。この構成は、従来型の自動運転システムで分離されていた認識、予測、判断、制御を個別のソフトウェアモジュールとして実装するモジュール構造ではない。認識、予測、判断、制御は、単一のAIモデルの内部機能として統合され、最終的には車両操作指示として現れる。

一方で、E2E AIを通常走行の中核に用いることは、故障時の安全停止までAI任せにすることを意味しない。カメラ、SoC、ECU、通信、電源、制動、操舵などの失陥は、E2E AIの通常機能で吸収するのではなく、**通常走行用ECUとMRC用ECUからなる異種ECU冗長**による多重系ADS、安全監視、縮退制御、MRCによって扱う。これにより、シリコン・ファームウェア・OS・AIモデル・systematic failure・センサ入力レベルの共通原因故障（CCF）を設計上排除する。

本書は、このTuringの安全設計思想を、安全論証として一貫した形で示すものである。

## 1.2 本書の目的

本書の目的は、TuringのL4自動運転車両について、以下を明確にすることである（第2版で9項目化）。

1. 何を安全と主張するのか

2. その主張をどのようなシステム構成で支えるのか（通常走行用ECU＋MRC用ECUの異種ECU冗長を含む）

3. E2E AIをどの範囲で安全論証するのか

4. ハードウェア失陥をどのように扱うのか

5. 一系統失陥時にMRC系ADSがどのようにMRCへ移行するのか

6. 遠隔監視をどのように位置づけるのか（特定自動運行制度との接続を含む）

7. ISO規格・UN規則・国内法令をどの目的で参照するのか（対応の四区分）

8. サイバーセキュリティおよび責任分界（技術・型式指定）をどのように構成するか

9. 今後どの衛星文書および段階的社会実装（Phase 1〜3）で詳細化していくか

# 2. 対象システムの概要

## 2.1 対象とするL4自動運転システム

本書が対象とするシステムは、定義されたODD内において、運転者の介入を前提とせずに走行するL4自動運転システムである。

このシステムは、通常走行時にはE2E AI（通常走行用E2E AI）が通常走行用カメラ群7台の画像を入力として舵角指示および加速度指示を生成する。失陥時には、通常走行用ECUとMRC用ECUからなる異種ECU冗長による多重系ADSと安全監視系によりMRCモードへ遷移し、MRC用ECU上のMRC専用E2E AIがMRC専用カメラ1台を入力として、車両単独でリスク最小状態を達成する。

## 2.2 システム構成の基本思想（異種ECU冗長を中核とする）

TuringのL4自動運転システムは、以下の構成を前提とする。本構成の中核思想は、通常走行ADSとMRC系ADSを**異種ECU冗長**として実装することである。すなわち、通常走行用ECU（高性能AI処理向け）とMRC用ECU（MRCタスク十分性能）を、異なる世代・異なる物理設計・異なる製造マスクのシリコンとして配置することで、シリコン・ファームウェア・OS・AIモデル・systematic failure・センサ入力レベルの共通原因故障（CCF）を設計上排除する。

| **構成要素** | **役割** |
| --- | --- |
| 通常走行用E2E AI | 通常走行用カメラ群7台の画像を入力し、舵角指示・加速度指示を出力する通常走行の中核 |
| MRC専用E2E AI | MRC専用カメラ1台を入力し、衝突回避と同一車線内安全停止に特化した推論を行う |
| 通常走行用ECU | 通常走行用E2E AIを実行する高性能AI処理向け計算機ECU |
| MRC用ECU | MRC専用E2E AIを実行する計算機ECU。通常走行用ECUと異種シリコンで構成 |
| 通常走行ADS | 通常走行用ECU上で通常走行用E2E AIを実行し、車両制御指令を生成する系統 |
| MRC系ADS | MRC用ECU上でMRC専用E2E AIを実行し、通常走行ADS失陥時にMRCモードへ遷移する系統 |
| 安全監視系 | ADS出力、システム状態、入力品質、車両応答、ODD逸脱を監視する |
| MRC制御機能 | 通常走行を継続せず、車両をリスク最小状態へ移行させる |
| 車両制御I/F | 制動、操舵、駆動遮断、ハザード、外向けHMI等を実行する |
| ログ・記録系 | E2E AI出力、失陥検知、フェイルオーバー、MRC遷移、車両状態をUN-R 157 DSSAD要件と整合する形で記録する |
| 遠隔監視・運用支援 | MRC後の状態確認、乗客対応、現場対応、回収、記録保全を支援する |

異種ECU冗長アーキテクチャの詳細は第9.4節および第9.10節で、カメラ構成（通常走行用7台 + MRC専用1台）の詳細は第10.2節で扱う。両系統の電源（LV1/LV2）、通信経路、時刻同期等の物理的分離、および残存する共通故障源への対処は第9.9節で整理する。

## 2.3 E2E AIの位置づけ

Turingの通常走行ADSでは、通常走行用カメラ群7台の画像を主入力とし、舵角指示および加速度指示を出力するE2E AIを用いる。

このE2E AIでは、歩行者、車両、信号、車線、道路構造、横断歩道、交通ルール、他者の挙動、停止・進行・操舵・減速といった要素が、単一モデルの内部表現として統合的に扱われる。

したがって、Turingのソフトウェア安全論証では、「認識モジュール」「予測モジュール」「判断モジュール」「制御モジュール」の個別正しさを主たる論証対象にしない。

代わりに、以下を論証の中心に置く。

*ODD内の画像入力に対して、E2E AIが安全な舵角指示および加速度指示を出力し、その結果として車両が安全な閉ループ挙動を取ること。*

なお、MRC系ADSにはMRC専用E2E AIを別途搭載する。MRC専用E2E AIは、MRC専用カメラ1台を入力として、衝突回避と同一車線内安全停止に特化した推論を行う。通常走行用E2E AIとMRC専用E2E AIは、モデル構造、学習データ、開発チーム、デプロイ経路を分離し、AIモデル由来のCCFを設計上排除する。

## 2.4 多重系ADSの位置づけ（通常走行用ECU＋MRC用ECUの異種ECU冗長）

TuringのL4安全論証では、ADSを多重系として、**通常走行用ECUとMRC用ECUからなる異種ECU冗長**で構成する。両ECUは、シリコン世代、物理設計、製造マスク、AIモデル、ファームウェア、OS、開発チームのいずれにおいても異なる構成を持ち、共通原因故障（CCF）を設計上排除することを基本方針とする。

多重系ADSの目的は、一系統失陥後に通常走行を継続することではない。一系統が失陥した場合でも、MRC系ADS（MRC用ECU + MRC専用カメラ1台）が自動的に制御権限を引き継ぎ、MRCモードへ遷移し、車両単独でリスク最小状態に到達することである。

したがって、多重系ADSの安全主張は以下である。

*一系統のADSが失陥した場合でも、もう一系統が異種ECU冗長によりCCF排除性を確保した上でMRCモードへ遷移し、通常走行を継続せず、車両を安全側へ移行させる。*

異種ECU冗長アーキテクチャの詳細、Cascading Failure・Latent Faultを含む共通原因故障の取扱い、両系統の電源・通信・時刻同期の物理的分離は、第9章で具体化する。

## 2.5 遠隔監視の位置づけ

遠隔監視は、MRC成立の一次安全手段ではない。

通信断、遠隔監視断、オペレータ不在であっても、車両は単独で故障検知、フェイルオーバー、MRC遷移、停止後の状態保持を行う必要がある。

遠隔監視は、MRC後の状態確認、乗客対応、現場対応、回収、記録保全を支援する運用手段として位置づける。日本国内における特定自動運行制度との関係（特定自動運行主任者の役割、遠隔監視装置の要件等）は、第12章で整理する。

# 3. 用語と前提条件

## 3.1 L4

本書におけるL4とは、定義されたODD内において、運転者が動的運転タスクを引き継ぐことを前提とせず、ADSが運転タスクを担う自動運転を指す。

## 3.2 ADS

ADSとは、自動運転システムを指す。本書では、E2E AI、計算機、センサ入力、安全監視、MRC制御、車両制御I/Fとの接続を含むシステムとして扱う。

## 3.3 E2E AI

E2E AIとは、カメラ画像等を入力し、舵角指示および加速度指示を出力するAIモデルを指す。

本書では、認識、予測、判断、制御を個別モジュールとして分けず、単一モデルの内部機能として統合的に扱うAIを想定する。

## 3.4 ODD

ODDとは、Operational Design Domain、すなわち自動運転システムが作動することを前提とする走行環境条件である。

ODDには、地理的条件、道路種別、速度域、交通環境、天候、照度、視界、路面状態、通信条件等を含む。

## 3.5 通常走行モード

通常走行モードとは、ハードウェアおよびシステムが健全であり、ODD内でE2E AIが舵角指示および加速度指示を生成し、目的地または経路に沿って走行する状態である。

## 3.6 MRCモード

MRCモードとは、失陥、機能低下、ODD逸脱等により通常走行を継続すべきでない場合に、車両をリスク最小状態へ移行させるための制御状態である。

MRCモードでは、目的地への走行継続ではなく、安全側への遷移を優先する。

## 3.7 MRM / MRC

MRMはMinimal Risk Manoeuvreを指し、リスクを最小化するための操作または一連の挙動を意味する。

MRCはMinimal Risk Conditionを指し、MRMの結果として到達すべきリスク最小状態を意味する。

本書では、必要に応じて、MRCを「リスク最小状態」または「安全側停止状態」として扱う。

## 3.8 通常走行ADS

通常走行ADSとは、通常走行時に主としてE2E AIを実行し、車両制御指令を生成するADS系統である。

## 3.9 MRC系ADS

MRC系ADSとは、通常走行ADSに失陥が生じた場合に、自動的に起動または制御権限を引き継ぎ、MRCモードへ遷移するADS系統である。

MRC系ADSの目的は、通常走行を継続することではなく、MRCを達成することである。

## 3.10 安全監視系

安全監視系とは、ADS出力、入力品質、システム状態、車両応答、ODD逸脱等を監視し、不安全な状態を検知した場合にMRC遷移を要求する機能または構成要素である。

## 3.11 フェイルオーバー

フェイルオーバーとは、通常走行ADSに異常が発生した場合に、MRC系ADSが自動的に制御権限を引き継ぐ、またはMRC制御を開始する状態遷移を指す。

本書では、フェイルオーバー後は通常走行へ自動復帰せず、MRCモードへ固定することを基本方針とする。

## 3.12 ハードウェア失陥

ハードウェア失陥とは、カメラ、SoC、ECU、通信、電源、制動、操舵、駆動、車両制御I/F等の故障、性能低下、異常動作を指す。

## 3.13 カメラ健全性

カメラ健全性とは、単に映像が入力されていることではない。MRCまたは通常走行に必要な品質、連続性、同期、露出、汚損状態、伝送完全性等を満たしていることを指す。

## 3.14 DSSAD相当ログ

DSSAD相当ログとは、自動運転システムの状態、E2E AI入力・出力、失陥検知、フェイルオーバー、MRC遷移、車両制御指令、車両状態量等を、事後に再現可能な形で記録するログを指す。

## 3.15 通常走行用ECU

通常走行用ECUとは、通常走行ADSにおいて、通常走行用E2E AIを実行する高性能AI処理向けの計算機ECUである。本書では実装ECUの具体名を扱わず、機能名「通常走行用ECU」のみを用いる。

## 3.16 MRC用ECU

MRC用ECUとは、MRC系ADSにおいて、MRC専用E2E AIを実行する計算機ECUである。MRCタスク（同一車線内安全停止）に十分な性能を備える。本書では実装ECUの具体名を扱わず、機能名「MRC用ECU」のみを用いる。

## 3.17 MRC専用カメラ

MRC専用カメラとは、MRC系ADSがMRC実行のために用いる、MRC用ECUに直結された前方広角カメラ1台を指す。通常走行用ECUは本カメラを使用しない（通常走行用カメラ群とは物理・電気・伝送系で完全独立とする）。

## 3.18 特定自動運行主任者（第2版で追加）

特定自動運行主任者とは、道路交通法に定める特定自動運行の枠組みにおいて、運行計画ごとに選任され、運行中の安全確保に責任を負う者を指す。

本書では、特定自動運行主任者を、MRC成立後の状態確認、乗客対応、現場対応、当局・関係機関への通報等を担う運用上の役割として扱う。MRC成立そのものは車両単独で完結することを基本とし（4.8、12.2参照）、特定自動運行主任者はMRC成立を支える一次安全手段としては位置付けない。具体的な選任要件、配置、業務範囲は衛星文書「特定自動運行計画書」で確定する。

## 3.19 DSSAD（UN-R 157）（第2版で追加）

DSSADとは、Data Storage System for Automated Drivingの略であり、UN-R 157が要求する自動運転システム作動状態の記録装置を指す。

UN-R 157はL3 ALKS向け規則であり、L4向け規則整備はWP.29 GRVAで議論中である。本書では、UN-R 157のDSSAD要求項目（作動／解除、運転者引継ぎ要求、Override、MRM、システム失陥、保存期間等）を、L4特有の要件（運転者非前提、車両単独MRC）に読み替えて参照する。具体的な記録項目、保存期間、提出インタフェースは衛星文書「ログ項目定義書」および第13章で具体化する。

本書において「DSSAD相当ログ」（3.14）と「DSSAD（UN-R 157）」は、目的を共有しつつも以下の関係にある。

- DSSAD相当ログ（3.14）：本書の安全論証における事後再現性確保のためのログ概念
- DSSAD（UN-R 157）（3.19）：UN-R 157が要求する規則上の記録装置

両者は対応付けて管理し、Appendix EおよびAppendix Fでマッピングする。UN-R 157のDSSAD要求項目への対応は、第13章および衛星文書「ログ項目定義書」で具体化する。

## 3.20 Triggering Condition（SOTIF概念の借用語彙）（第2版で追加）

Triggering Conditionとは、ISO 21448 / SOTIFが提供する概念であり、意図された機能の性能不足または仕様不足が、不安全な車両挙動として顕在化する条件を指す。

本書はISO 21448への形式的準拠を目的としない（6.4参照）。一方で、E2E AIの性能限界を記述する共通語彙としてTriggering Conditionの概念を借用する。

- ODD境界付近で性能限界が顕在化する条件（第7.4節）
- E2E AIが安全側挙動を逸脱しうる条件（第8.5節）
- 不確実な場面、未知・稀シナリオの記述語彙（第8章、第21章）

具体的なTriggering Conditionの列挙は、AI Safety Caseおよび衛星文書「お台場ODD定義書」で行う。本書本文ではTriggering Conditionを語彙として用い、個別clauseへの適合主張は行わない。

## 3.21 Cascading Failure / Latent Fault（ISO 26262）（第2版で追加）

Cascading Failureとは、ISO 26262の用語であり、一つの要素の故障が、他の要素の故障または機能喪失を引き起こす波及的故障を指す。

Latent Faultとは、ISO 26262の用語であり、それ単独では直ちに安全機能を喪失させないが、別の故障と組み合わさることで安全機能を喪失させうる潜在故障を指す。

本書では、多重系ADSおよび異種ECU冗長アーキテクチャの安全論証において、これら二つの概念を以下の意味で用いる（第9.10節、第6.3節）。

- Cascading Failure：通常走行系の故障がMRC系に波及しないことを論証する観点
- Latent Fault：MRC系の準備状態（待機中の機能）に潜在故障が蓄積していないことを論証する観点

定量目標（SPFM、LFM、PMHF等）は本書では確定せず、衛星文書「車両要求仕様書」および各ECUのSafety Manualで確定する。

## 3.22 Positive Risk Balance（第2版で追加）

Positive Risk Balance（以下「PRB」）とは、自動運転車両のリスクを、同一ODDにおける人間運転者の事故率と比較し、これを下回ることを許容基準の一つとする考え方を指す。

本書では、PRBを、許容しきい値の定義における比較ベンチマークの基本型として用いる（4.2、4.6、21.1参照）。

- 許容しきい値は、絶対値のみで定義せず、人間運転者の事故率を比較ベンチマークの一つとする
- 比較対象は同一ODDにおける人間運転者の事故率、規制基準、運用上の受入基準を組み合わせる
- 単一指標に依存せず、重大事故率、致死事故率、ハザード別発生率等を併用する

具体的なベンチマーク選定、データソース、許容しきい値の数値は、後続の証拠文書（HARA、運用後監視文書）で具体化する。

# 4. 最上位安全主張

## 4.1 最上位安全主張

Turingの最上位安全主張は、以下である。

*TuringのL4自動運転車両は、定義されたODD内において、車両起因の重大事故および重大インシデントの発生率が、別途定める許容しきい値を下回ることを主張する。*

本主張は、以下の2つのサブゴールにより支える。

(a) 通常走行リスクが許容範囲内であること:ハードウェアが健全な状態において、E2E AIによる閉ループ車両挙動が安全に成立し、通常走行に起因する重大事故率が許容しきい値を下回る。

(b) 失陥時のMRC成立性:ADSまたは車両構成要素に失陥が生じた場合、車両は通常走行を継続せず、多重系ADSと安全監視によりMRCモードへ遷移し、車両単独でリスク最小状態に到達する。これにより失陥起因の重大事故率が許容しきい値を下回る。

(a)および(b)の主張は、運用後監視、モデル・ソフトウェア更新管理、未知シナリオの取込、共通原因故障（CCF）・Cascading Failure・Latent Faultの継続評価等により、運用期間を通じて漸近的に補強する。これは独立したサブゴールではなく、(a)(b)主張の信頼性を維持する支援的論証として扱う（4.7参照）。

本書は、安全主張の構造、許容しきい値の定義方針、サブゴールを支える設計および証拠の枠組みを示す。許容しきい値の数値、合格基準、検証手順は本書では確定せず、後続の証拠文書で具体化する。

## 4.2 許容しきい値の定義方針

本書では、4.1に示した許容しきい値そのものの数値を確定しない。これは、本書が安全論証の上位文書であり、定量基準の確定は後続証拠文書の役割であるためである。

ただし、しきい値の定義方針は以下とする。

1. 単一指標に依存しない。重大事故率、致死事故率、ハザード別発生率など複数指標を併用する。

2. 比較ベンチマークを明示する。同一ODDにおける人間運転者の事故率、規制基準、または運用上の受入基準を参照する(Positive Risk Balance型を基本とする)。

3. ハザード別に分解する。少なくとも、衝突、車線逸脱、信号・標示違反、MRC失敗、ODD逸脱、不適切な車線変更を区分する。

4. 走行時間統計のみによる実証可能性を前提としない。構造的安全論証、シナリオベース評価、故障注入、運用後監視を組合せ、しきい値の妥当性を漸近的に確認する。

5. ODDおよび運用条件を明示する。許容しきい値は対象ODDと運用条件に紐づく。

数値しきい値、ベンチマーク選定、ハザード分類、運用後監視指標は、後続の証拠文書で具体化する。

なお、4.2の「比較ベンチマークを明示する」原則は、Positive Risk Balance（3.22）の考え方を基本型として用いることを意味する。同一ODDにおける人間運転者の事故率、規制基準、運用上の受入基準を組み合わせ、単一指標に依存しない構成を取る。

## 4.4 安全主張を支える基本前提

4.1の安全主張は、以下の前提に基づく。

1. ODDが明確に定義されていること（第7章、お台場ODD定義書）

2. 通常走行時には、ハードウェアおよび車両制御系が健全であること

3. E2E AIが、ODD内の画像入力に対して安全な舵角指示および加速度指示を生成すること

4. E2E AIの出力が、安全監視系により監視されること

5. 一系統のADSが失陥した場合、MRC系ADSがMRCモードへ遷移すること（異種ECU冗長アーキテクチャを前提とする。第9.4節）

6. MRC関連機能が単一故障で同時に喪失しないよう設計されること（Cascading Failure / Latent Faultの観点を含む。第9.10節、第3.21節）

7. 遠隔監視に依存せず、車両単独でMRCを完了できること（4.8、第12.2節）

8. 失陥、フェイルオーバー、MRC遷移、停止結果が記録されること（DSSAD相当ログ、第13章、第3.14節、第3.19節）

9. ソフトウェアおよびE2E AIモデルの更新が管理され、更新前後で安全主張が維持されること（rolling update中の系間整合を含む。第6.7節、モデル更新管理書）

10. サイバー起因の安全影響が独立に管理されること（第14章、Cybersecurity Case、第6.8節）

## 4.5 サブゴール(a) 通常走行リスクが許容範囲内であること

サブゴール(a)の主張は、以下である。

*ハードウェアが健全であり、車両がODD内にある場合、E2E AIはカメラ画像等の入力に基づき、安全な舵角指示および加速度指示を生成し、車両として安全な閉ループ挙動を実現する。これにより通常走行起因の重大事故率が許容しきい値を下回る。*

ここでいう安全な閉ループ挙動には、以下を含む。

車線および道路端に対する横方向安全

前方車両、歩行者、自転車、二輪車に対する縦方向安全

信号、一時停止、横断歩道、速度制限等への対応

不確実な状況での減速または停止

急な舵角変化、急加減速、不要な挙動の抑制

ODD境界付近での安全側遷移

サブゴール(a)を支える設計および評価の枠組みは、第8章で具体化する。AI Safety Requirementsの導出は第8.8節およびAI Safety Caseで担う。

## 4.6 サブゴール(b) 失陥時のMRC成立性

サブゴール(b)の主張は、以下である。

*ADS、センサ、計算機、通信、電源、制動、操舵等に失陥が生じた場合、TuringのL4自動運転車両は、E2E AIの通常機能で失陥を吸収しようとせず、失陥を検知し、安全監視および多重系ADSによりMRCモードへ遷移する。MRCモードでは通常走行判断を制限し、現在車線内または直近の安全停止可能位置に停止することを基本とする。MRC完了後は自動再発進を制限し、遠隔監視または現場対応を待つ。これにより失陥起因の重大事故率が、停止後のリスクを含めて許容しきい値を下回る。*

失陥時の目的は、走行継続ではなく、車両を安全側に遷移させ、リスク最小状態に到達させることである。

ここでいうリスク最小状態には、停止後の被追突、降車乗客、現場対応中の周囲交通等に対するリスクが許容範囲内に保たれることを含む。

MRCモードで原則として制限する行動は、以下である。

車線変更

追越し

後退

積極的な合流

通常右左折

不確実な交差点進入

不確実な横断歩道通過

サブゴール(b)を支える設計および評価の枠組みは、第9章から第11章で具体化する。第9.4節の異種ECU冗長アーキテクチャ、第9.10節の共通原因故障（Cascading Failure / Latent Faultを含む）の扱いを中核とする。

サブゴール(b)の許容しきい値の根拠は、Positive Risk Balance（3.22）を基本型とする。

具体的には、以下の考え方を採用する。

- 失陥時のMRC成立性に関する許容しきい値は、絶対値のみで定義せず、同一ODDにおける人間運転者の事故率（重大事故率、致死事故率、ハザード別発生率）を比較ベンチマークとする
- 停止後の被追突、降車乗客、現場対応中の周囲交通等のリスクも、人間運転者が同等失陥状態（例：パンク停止、エンスト停止）に置かれた場合のリスクと比較する
- MRC成立率、MRC遷移時間、MRC位置選定の妥当性等の運用後監視指標を、Positive Risk Balanceの確認手段として用いる（第21章、運用後監視文書）

具体的なベンチマーク数値、データソース、合格基準は、後続の証拠文書（HARA、車両要求仕様書、運用後監視文書）で確定する。

## 4.7 安全主張の信頼性維持

4.1のサブゴール(a)および(b)は、設計、解析、評価により成立を主張するが、その根拠は完全ではない。具体的には、以下のような不確実性が残る。

未知・稀シナリオに対するE2E AIの挙動

学習データ偏り、ロバスト性、入力品質感度等のAI体系的エラー

複数系統に同時影響する共通原因故障

モデル更新およびソフトウェア更新による安全主張の変動

運用条件、交通環境、道路状況、規制の経時変化

これらに対しては、安全主張をサブゴールとして追加するのではなく、(a)および(b)主張の信頼性を維持する支援的論証として、以下の活動を継続的に行う。

| **活動** | **目的** |
| --- | --- |
| 運用後監視 | 実走行データに基づきサブゴール(a)(b)の前提と挙動を継続確認する |
| 異常事例の収集と反映 | MRC事例、ヒヤリハット、苦情、近接事象を分析し設計と評価に反映する |
| モデル更新管理 | 更新前後でサブゴール(a)が維持されることを確認する(ISO 24089) |
| ソフトウェア更新管理 | 通常走行系、安全監視系、MRC系の更新を管理し、サブゴール(b)を維持する |
| 共通原因故障の継続評価 | 設計時に想定した共通原因の分離が運用上も成立しているかを確認する |
| ODDおよび運用条件の見直し | ODD逸脱事象や運用環境の変化に応じてODD・前提条件を更新する |

本活動の成果は、安全論証の更新としてフィードバックされる（第21章「残存リスクと今後の更新」、Phase 1〜3の段階別更新方針を参照）。

## 4.8 遠隔監視に関する補足主張

遠隔監視に関する補足主張は、以下である。

*遠隔監視は、MRC成立の一次安全手段ではない。通信断、遠隔監視断、オペレータ不在であっても、車両は単独でMRCへ移行し、リスク最小状態に到達する。*

遠隔監視は、MRC後の状態確認、乗客対応、現場対応、回収、記録保全を支援する運用手段として扱う。特定自動運行主任者（3.18）は、運用上の役割としてこれらを担うが、MRC成立そのものを支える一次安全手段としては位置付けない。詳細は第12章および衛星文書「特定自動運行計画書」で具体化する。

# 5. 安全論証の全体構造

## 5.1 安全論証を分けて考える理由

TuringのL4安全論証では、通常走行時のE2E AI安全と、失陥時のMRC安全を明確に分ける。

この分離は重要である。なぜなら、通常走行時のAI性能と、ハードウェア故障時の安全停止は、性質の異なるリスクだからである。

通常走行時には、ハードウェアが健全であることを前提に、E2E AIが道路環境、交通参加者、交通ルール、不確実性に対して安全な車両挙動を生成できるかを扱う。これはISO/PAS 8800の枠組みとSOTIF概念借用語彙（Triggering Condition、4象限、Performance Limitation等）で整理する（第6.4節、第6.5節）。

一方、失陥時には、カメラ、SoC、ECU、通信、電源、制動、操舵等の故障により、通常走行を継続すべきでない状態が発生する。この場合、E2E AIの通常機能で問題を吸収するのではなく、車両アーキテクチャとしてMRCへ遷移する。これはISO 26262の枠組み（Cascading Failure、Latent Fault、診断カバレッジを含む）で整理する（第6.3節、第3.21節）。

両者の接続は、安全監視系（3.10）と多重系ADS（第9章）が担う。E2E AI出力に対する安全監視は通常走行領域で機能し、失陥検知・フェイルオーバーは失陥領域で機能する。

## 6.2 ISO/TS 5083の使い方

ISO/TS 5083:2025は、車両に統合されたADSの安全を達成し、示すためのガイダンスであり、設計、検証・妥当性確認、市場投入後活動を含み、ISO/SAE PAS 22736で定義されるレベル3およびレベル4 ADS機能を対象としている。

Turingの安全論証では、ISO/TS 5083を、ADS全体の安全論証の上位フレームとして用いる。

| **用途** | **内容** |
| --- | --- |
| 最上位安全主張 | L4 ADS全体として何を安全と主張するか |
| 安全ケース構造 | 通常走行、安全監視、失陥時MRC、運用後対応をどう接続するか |
| 設計と証拠の対応 | 安全要求、アーキテクチャ、証拠をどう紐づけるか |
| 運用後活動 | ログ、MRC事例、更新管理、継続的改善の位置づけ |

本書では、ISO/TS 5083を安全論証全体の骨格として参照する。詳細な対応表はAppendix Eで具体化する。

## 6.3 ISO 26262の使い方

ISO 26262は、道路車両の安全関連E/Eシステムを対象とする機能安全の規格である。ISO 26262の枠組みは、E/Eシステムの故障に起因する危険を管理するために用いられる。

Turingの安全論証では、ISO 26262を、ハードウェア失陥安全論証の基礎として用いる。第2版では、Cascading Failure、Latent Fault、診断カバレッジの考え方を明示的に取り込む。

| **対象** | **ISO 26262の使い方** |
| --- | --- |
| ECU / SoC異常 | 計算系のハング、停止、遅延、異常出力を失陥として扱う |
| 電源異常 | 単一電源故障でMRC関連機能が同時喪失しないことを整理する |
| 通信異常 | 通信断、遅延、破損、時刻同期ずれを失陥として扱う |
| 制動・操舵・駆動 | 車両制御系の失陥時に縮退または安全停止できることを整理する |
| 多重系ADS | 一系統失陥時にMRC系がMRCへ移行する構造を整理する |
| 共通原因故障(CCF) | 多重系が同時に失われるリスクを整理する(第9.10節) |
| **Cascading Failure** | 一系統の故障が他系統へ波及するリスクを整理する(第9.10節) |
| **Latent Fault** | 直ちに機能喪失を起こさないが、別の故障と組み合わさって安全機能を喪失させる潜在故障の管理(第9.10節) |
| **診断カバレッジ** | 通常走行時の安全監視、MRC系自己診断、車両ECU群のハートビート、HW Built-In Self-Test等を組合せて達成 |

ISO 26262 Part 5のSPFM(Single Point Fault Metric)、LFM(Latent Fault Metric)、PMHF(Probabilistic Metric for Hardware Failures)の定量目標は、衛星文書「車両要求仕様書」および各ECUのSafety Manualで確定する。本書本文ではこれらを観点参照として位置付ける。

本書では、E2E AIの性能不足と、E/Eシステムの故障を混同しない。カメラフリーズ、SoCハング、電源低下、通信断、制動系故障等は、ISO 26262の考え方に基づき、ハードウェア失陥安全として扱う。

## 6.4 ISO 21448 / SOTIFの扱い(概念借用・形式的準拠は目的としない)

ISO 21448:2022は、ISO 26262で扱う故障を対象外とし、意図された機能の仕様不足または性能不足に起因するリスクを対象とする規格である。

**本書はISO 21448への形式的準拠を目的としない**。すなわち、SOTIFのプロセス(機能仕様起点のHazard Analysis、Triggering Conditionの体系的同定、SOTIF V-cycleの工程適用、SOTIF Assessment Reportの独立成果物としての整備等)を、本書の安全活動として実施することは宣言しない。

一方で、本書は**SOTIFが提供する以下の概念を、E2E AIの性能限界と残存リスクを論証するための共通語彙として借用する**。これらの概念は本書独自の論証構造に組み込まれており、ISO 21448個別clauseへの適合主張ではない。

| 借用する概念 | 本書での使い方 |
| --- | --- |
| Triggering Condition | E2E AIの性能限界が顕在化する条件を記述する語彙として用いる(第7.4節、第8.5節)。具体例はAI Safety Caseおよびお台場ODD定義書で列挙する |
| Known Safe / Known Unsafe / Unknown Safe / Unknown Unsafe(4象限) | E2E AIの汎化能力と検証カバレッジを論証する構造として用いる(第8章、第21章) |
| Validation of Verification Means | シミュレータと実走行の相関、検証環境の妥当性を論証する考え方として用いる(第8章、第20章) |
| Performance Limitation / Functional Insufficiency | E2E AIに固有の機能限界を記述する語彙(第8章、第21章) |
| Reasonably Foreseeable Misuse | 乗員misuseの扱いを記述する語彙(第14章、内向けHMI仕様書で具体化) |

借用の理由は以下である。

第一に、E2E AIの性能限界は、ISO 26262の故障モデルでは記述しきれない。SOTIFが提供する「故障していないが性能不足で危ない」という語彙は、E2E AIの安全論証で必要となる。

第二に、JARI、国交省、NALTEC、UN-R 157 GRVA、ISO/PAS 8800、OEMの安全文化は、SOTIF語彙を共通言語として用いる。本書がこの語彙を借用することで、対外説明・協議のコストを最小化できる。

第三に、ISO/PAS 8800がSOTIF概念の上に構築されている以上、PAS 8800を参照する本書(第6.5節)は、SOTIF語彙を間接的に必要とする。

借用しない要素は以下である。

| 借用しない要素 | 理由 |
| --- | --- |
| SOTIF V-cycleの形式的工程適用 | 画像入力から舵角・加速度を直接出力するE2E AIに対して、SOTIFが前提とする機能分解(Triggering Condition→Hazardous Behavior)の粒度がアーキテクチャと一致しない |
| SOTIF Assessment Reportの独立成果物 | 本書および衛星文書(AI Safety Case、HARA、検証文書群)で同等の論証を提供する |
| ISO 21448 clauseへの適合宣言 | 本書の方針(6.1)に従い、形式的準拠は目的としない |

本書の立場として、ISO 21448を「軽く参照」することは行わない。「**概念を借用するが、形式的準拠は目的としない**」という方針を明示する。これにより、対外的に「準拠したのに満たしていない」という指摘を避けつつ、SOTIF語彙の対外説明力を確保する。

論証分担は以下のとおりである。

| 対象 | 本書での扱い | 主たる規格・参照 |
| --- | --- | --- |
| E2E AIの性能限界 | 本書第8章 + AI Safety Case | ISO/PAS 8800、SOTIF概念借用 |
| ODD境界 | 本書第7章 + ODD定義書 | ISO 34503 ODD taxonomy、SOTIFのODD境界・Triggering Condition語彙借用 |
| 不確実な場面、未知・稀なシナリオ | 本書第8章 + AI Safety Case + 検証文書 | ISO/PAS 8800、ISO 34502系、SOTIF 4象限の語彙借用 |
| 安全側挙動 | 本書第4章、第5章、第8章、第11章 | 本書独自構造 |
| 非AIセンサの性能限界 | 入力品質劣化として本書第10章 + AI Safety Case、HW失陥として本書第10章 + 車両要求仕様書 | ISO/PAS 8800 + ISO 26262 |
| Misuse(乗員) | 本書第14章 + 内向けHMI仕様書 | SOTIFのReasonably Foreseeable Misuse語彙借用 |

# 7. ODDの定義と扱い

## 7.1 ODDを定義する目的

ODD（Operational Design Domain）は、TuringのL4自動運転車両が通常走行モードで作動する前提条件を定めるものである。

ODDを明確に定義する理由は、E2E AIの安全論証、失陥時のMRC設計、遠隔監視の運用範囲、残存リスクの判断を、すべて同一の前提に基づいて行うためである。ODDが曖昧なままでは、E2E AIがどの環境まで安全責任を持つのか、どの条件でMRCへ遷移すべきか、運用上どのような制約を置くべきかが収束しない。

第2版では、ODDをISO 34503のODD taxonomy、ISO 21448が借用語彙とするODD境界・Triggering Condition、UN-R 157 5.2 ODD要件と整合する形で定義する。形式的準拠は目的としない（第6.4節）が、対外説明・協議の共通言語として上記語彙を用いる。

ODDの定義は本書では概要にとどめ、詳細は衛星文書「お台場ODD定義書」で確定する。本書本体での主張は、(i) ODDがL4安全論証の前提条件であること、(ii) ODD内/ODD境界/ODD外を明確に区別して扱うこと、(iii) ODD境界およびODD逸脱は安全監視系がリアルタイムに判定し、必要に応じてMRC遷移を要求すること、の三点である。

## 7.2 ODDに含める条件

TuringのODDは、ISO 34503のODD taxonomy（Scenery / Environmental Conditions / Dynamic Elements）を参照しつつ、本書のシステム構成（E2E AI＋多重系ADS＋MRC専用カメラ）に整合する形で、以下の項目を最低限定義する。

| **分類** | **ISO 34503相当** | **定義する内容** |
| --- | --- | --- |
| 地理的条件 | Scenery / Zones | 走行可能エリア（ジオフェンス）、地図範囲、サービス提供範囲 |
| 道路条件 | Scenery / Drivable area | 道路種別、車線数、車線幅、交差点種別、歩道・路肩の有無、勾配・曲率 |
| 速度条件 | Operational | 通常走行時の速度範囲、MRC開始時に想定する速度域 |
| 交通環境 | Dynamic elements | 歩行者、自転車、二輪車、一般車両、商用車、駐停車車両、緊急車両の存在条件 |
| 交通制御 | Scenery / Traffic infrastructure | 信号、標識、道路標示、一時停止、横断歩道、規制標識 |
| 天候条件 | Environmental | 晴天、曇天、雨（雨量域）、霧（視程域）、雪、強風の扱い |
| 照度条件 | Environmental | 昼間、薄暮、夜間、逆光、トンネル出入口 |
| 路面状態 | Environmental | 乾燥、湿潤、積雪、凍結、反射、低摩擦 |
| 通信条件 | Connectivity | 遠隔監視通信の有無。ただしMRC成立の前提とはしない（第12章） |
| 特殊状況 | Operational | 工事、事故、警察・消防対応、道路閉鎖、イベント時規制 |

各項目の定量境界（例：雨量[mm/h]、視程[m]、照度[lx]、車速[km/h]）は本書では確定しない。これらは「お台場ODD定義書」および「AI Safety Case」で具体化する。

## 7.2.1 初期運用ジオフェンス（お台場・青海エリア）

TuringのL4自動運転車両の初期運用では、地理的条件として東京都港区台場および江東区青海の臨海副都心エリア（お台場・青海エリア）に閉領域型ジオフェンスを設定する。本ジオフェンスは通常走行モードに入ることが許容される地理的範囲を限定するものであり、E2E AIの安全論証、評価データ収集、シナリオ網羅、MRC設計の前提条件となる。

ジオフェンスの設定領域、選定意図、領域形状、ジオフェンス境界の運用、変更管理プロセス、座標およびポリゴン頂点等の詳細は、衛星文書「お台場ODD定義書」で一元管理する。本書では、安全論証における前提として、以下の概要のみを示す。

| **項目** | **概要** |
| --- | --- |
| 領域 | 東京都港区台場、江東区青海の一部を内包する閉ポリゴン |
| 含む主要施設 | お台場海浜公園、東京国際クルーズターミナル、日本科学未来館、青海ふ頭公園 等 |
| 除外する領域 | 首都高湾岸線本線、レインボーブリッジ、本土側道路、第六台場、有明地区 |
| 選定理由 | 道路構造・交通環境・信号配置が整理され、現実的な交通参加者が存在しつつ高速走行を排除できるため |

ジオフェンスの概略図はAppendix（初期運用ジオフェンス地図）に示す（参照：`appendix_geofence_odaiba_aomi.png`）。

ジオフェンス内部にいることはODD内であるための必要条件であるが、十分条件ではない。車両がジオフェンス内部にあっても、7.2に定義する天候、照度、視界、路面状態、通信条件、特殊状況等を満たさない場合、車両はODD外と判定し、通常走行モードを継続しない。ジオフェンス境界の安全監視系による扱いは7.4で示す。

ジオフェンスは固定値ではなく、運用、評価、エビデンスの蓄積に応じて、段階的に拡張または変更され得る。拡張・縮小・形状変更は変更管理プロセスを経て行い、本書（特に第8章、第11章、第21章）およびAI Safety Case、検証文書群への影響を再評価する。詳細は「お台場ODD定義書」で扱う。

## 7.3 ODD内通常走行

ODD内通常走行では、(i) ハードウェアが健全であること、(ii) E2E AIが入力画像から安全な舵角指示および加速度指示を出力できること、(iii) 安全監視系が異常を検知していないこと、の三点を前提とする。

ただし、ODD内であっても不確実性が高い場面や安全側判断が必要な場面では、E2E AIまたは安全監視系が減速、停止、またはMRC遷移を選択できる必要がある。すなわち、ODD内であることは「通常走行を継続することが許容される」ことを意味するが、「通常走行を継続しなければならない」ことを意味しない。安全側判断によるMRC遷移は、ODD内であっても発生し得る。

## 7.4 ODD境界

ODD境界とは、通常走行を継続できる条件と、通常走行を継続すべきでない条件の境界である。ODD境界の判定には、(a) 静的境界（ジオフェンス境界、地図上の対象外領域）と、(b) 動的境界（環境条件・車両状態・E2E AI性能限界の顕在化）の二種類がある。

第2版では、動的境界の判定にあたり、SOTIF（ISO 21448）が提供するTriggering Conditionの語彙を借用する（第6.4節）。Triggering Conditionは、E2E AIまたは多重系ADSの性能限界が顕在化する条件であり、ODD境界として安全監視系が判定する対象となる。

代表的な機能限界トリガー条件の例を以下に示す。これらは網羅ではなく、対外説明・協議のための例示である。完全な列挙は「お台場ODD定義書」および「AI Safety Case」で行う。

| **カテゴリ** | **代表的なトリガー条件** | **想定される判定主体** |
| --- | --- | --- |
| 環境条件 | 雨量がODD定義の閾値を超える、霧で視程が想定下限を下回る、降雪・吹雪、強風 | 安全監視系（車両搭載センサ＋気象データ） |
| 照度条件 | 夜間照度がODD定義の閾値を下回る、強い逆光、トンネル出入口での急激な明暗変化 | 安全監視系（カメラ露出統計） |
| 視界条件 | カメラ画像がMRCまたは通常走行に必要な品質を満たさない（汚損、曇り、結露、白飛び、黒潰れ） | カメラ健全性監視（第10.5節） |
| 路面条件 | 凍結・積雪・冠水を検知、低摩擦兆候、想定外の路面段差 | 安全監視系、車両状態量 |
| 道路構造 | 想定外の道路構造（工事、突発的車線封鎖、地図不整合、信号機の異常） | E2E AIの不確実性検知＋安全監視系 |
| 地理境界 | ジオフェンス境界への接近・跨ぎ・逸脱、地図境界の到達 | 安全監視系（位置・地図照合） |
| 交通規制 | 交通規制により通常経路が使用できない、警察・消防対応中の現場、道路閉鎖 | E2E AIの認識＋運用情報 |
| 車両状態 | 車両状態量が通常走行に適さない（タイヤ空気圧、ブレーキ残存能力、電源残存容量、制御I/Fの劣化兆候） | 安全監視系、車両ECU |
| AI不確実性 | E2E AIが高い不確実性を申告する場面、Out-of-Distribution検知、補助出力の信頼度低下 | E2E AI内部＋安全監視系 |
| 通信条件 | 通信断・遅延・時刻同期ずれが安全関連通信の品質基準を下回る | 安全監視系（通信品質監視） |

ODD境界の判定主体、判定タイミング、判定アルゴリズム、定量閾値、ヒステリシスの扱いは、本書では確定しない。これらは「お台場ODD定義書」「AI Safety Case」「安全監視系設計書」で具体化する。

## 7.5 ODD逸脱時の基本挙動

ODDを逸脱した場合、車両は通常走行を継続しない。ODD逸脱時には、状況に応じて以下のいずれかまたは複数を行う。

1. 安全側に減速する
2. 通常走行判断を制限する（車線変更、追越し、合流の抑制）
3. MRCモードへ遷移する（第11章）
4. 現在車線内または直近の安全停止可能位置へ停止する
5. 停止後は自動再発進を制限する（第9.8節）
6. 遠隔監視または現場対応を待つ（第12章）

ODD逸脱の検知からMRC遷移までの時間制約、遷移中の車両挙動、停止位置選定の考え方は、第11章および「MRC設計書」で扱う。ODD逸脱事象のログ記録は第13章で扱う。

# 8. E2E AIによる通常走行安全

## 8.1 E2E AIの基本思想

Turingの通常走行ADSは、カメラ画像を主入力とし、舵角指示および加速度指示を出力するE2E AIを中核とする。

このE2E AIでは、従来の自動運転システムにおける認識、予測、判断、制御が、独立したソフトウェア部品として明示的に分離されているわけではない。歩行者、車両、信号、車線、横断歩道、道路構造、交通ルール、他者挙動、停止、進行、譲り、回避、操舵、減速といった要素は、単一モデルの内部表現として統合的に扱われる。

第2版では、本章の安全論証はE2E AIに固有の事情を踏まえつつ、ISO/PAS 8800（AI safety）を主要参照とする。SOTIF（ISO 21448）からはTriggering Condition、4象限（Known/Unknown × Safe/Unsafe）、Performance Limitation、Validation of Verification Means等の概念のみを語彙として借用する（第6.4節）。E2E AIの実体としての論証は衛星文書「AI Safety Case」が担い、本書本体では上位主張、安全側挙動カテゴリ、AI Safety Requirements導出方針、安全監視と限界条件を提示する。

## 8.2 入力と出力

E2E AIの入出力は、以下を基本とする。

| **区分** | **内容** |
| --- | --- |
| 主入力 | 通常走行用カメラ群（7台、第10.2節）からの複数視点画像 |
| 補助入力 | 過去フレーム、車速、操舵角、加速度、ヨーレート等の必要な車両状態量 |
| 主出力 | 舵角指示、加速度指示 |
| 補助出力 | 必要に応じた安全監視用情報、信頼度、不確実性推定、ODD境界判定の補助情報、Out-of-Distribution検知結果等 |

主入力・補助入力・主出力の信号定義、メッセージ形式、E2E protection、周期、許容遅延は衛星文書「VMC I/F仕様書」で確定する。

補助出力の有無や形式にかかわらず、安全論証の中心は、最終的な舵角指示および加速度指示と、それによる車両挙動である（第8.4節）。

## 8.3 認識・予測・判断・制御の扱い

Turingでは、認識・予測・判断・制御という言葉を、実装上のモジュール名としては扱わない。一方、安全説明上の観点としては使用する。

| **従来用語** | **Turingにおける扱い** |
| --- | --- |
| 認識 | 画像内の信号、歩行者、車両、車線、障害物等が、安全な操作出力に反映されること |
| 予測 | 他者の将来の動きや危険可能性が、減速、停止、余裕確保として出力に反映されること |
| 判断 | 止まる、進む、譲る、待つ、回避するという選択が、舵角・加速度指示として現れること |
| 制御 | 出力された舵角・加速度により、車両が安定して走行、減速、停止できること |

つまり、TuringのE2E AI安全論証では、内部に個別の認識・予測・判断・制御モジュールがあることを前提にしない。その代わり、画像入力に対する舵角・加速度出力と、その結果としての車両挙動を証拠化する（第8.4節）。

## 8.4 通常走行安全を車両挙動として説明する考え方

E2E AIの安全性は、モデル内部の解釈だけではなく、車両の閉ループ挙動として説明する。

たとえば、赤信号を認識しているかどうかを内部モジュールの出力で説明するのではなく、赤信号が画像入力に含まれる場面で、E2E AIが停止に必要な加速度指示を出し、車両が停止線手前で停止することを説明する。

歩行者についても同様である。Turingの通常走行ADSはE2E AIを中核とし、認識・予測・判断・制御を独立したモジュールとして分離していない（第8.1節、第8.3節）ため、歩行者認識スコアのような中間出力をモデルから取り出して安全性の根拠とすることはできない。代わりに、横断歩道付近で歩行者が存在または存在し得る場面において、車両が適切に減速または停止することを、画像入力と舵角・加速度出力、および結果としての車両挙動の対応により説明する。

この閉ループ挙動論証は、ISO/TS 5083のSafety Case構造、UL 4600のargumentation構造、ISO/PAS 8800のValidationの考え方と整合する。形式的準拠は目的としない（第6.1節）が、対外説明の構造として上記規格群と整合させる。

## 8.5 E2E AIに求める安全側挙動

E2E AIには、ODD内の各場面において安全側挙動を求める。本節では、安全側挙動を求めるカテゴリを定義し、各カテゴリの代表的な場面と挙動を例示する。

完全な場面カタログ、各場面に対する安全要件、試験シナリオ、合否基準は、本書では確定しない。これらは「お台場ODD定義書」「AI Safety Case」「検証シナリオ集」で具体化する。

安全側挙動を求めるカテゴリと代表例は、以下である。

| **カテゴリ** | **対象とする場面の例** | **代表的な安全側挙動** |
| --- | --- | --- |
| 静的障害物 | 停止車両、赤信号、停止線、路上落下物、工事帯、駐停車車両 | 十分な余裕をもって減速し、停止線または障害物の手前で停止する |
| 動的他者 | 歩行者、自転車、バイク、配送スクータ、他車両 | 減速および距離確保を行い、進路が予測しにくい相手に対しては速度を落とす |
| 不確実性 | 横断意図不確実、車線不明瞭、前方空間不確実、側方確認不十分、見通し不良 | 急操舵を避け、加速を抑制し、車線変更や右左折を抑制する |
| ODD境界 | ODD境界接近、ODD逸脱の予兆、地図不整合、GNSS劣化 | 通常走行の継続を避け、MRC遷移を許容する |
| 優先権・交通ルール | 右直関係、合流、譲り合い、一時停止、信号従動、横断歩道優先 | 他者の優先権を尊重し、合流・進入時に安全確認を行い、ルールに従う |
| 悪条件 | 雨、霧、夜間、逆光、強風、路面不良、低視認、低摩擦 | 速度低下、車間拡大、急操舵・急加速の抑制、必要に応じMRC遷移を許容する |
| 緊急対応 | 緊急車両、子供・高齢者の急な挙動、想定外飛び出し、後続車急接近 | 緊急車両に道を譲り、弱者の挙動に余裕を持ち、予備減速・回避操作を行う |

これらのカテゴリと挙動は、ISO/PAS 8800のロバスト性、ODD適合性、分布外評価および残存リスク主張に対応付けて扱う。SOTIF（ISO 21448）への形式的準拠は本書の目的ではない（第6.4節）ため、Triggering Conditionに相当する分析は、AI Safety Case内でISO/PAS 8800の枠組みに従って整理する。

カテゴリ横断の安全側挙動として、以下も求める。

- ODD内の制限速度・推奨速度を遵守すること
- 後続車との関係で過度な急減速を避けること（ただし前方衝突回避が優先する）
- 車間距離を路面・速度・天候に応じて確保すること
- E2E AIが想定外場面または高い不確実性を検知した場合、第8.7節に従いMRC遷移を許容すること

これらの安全側挙動が達成されているかは、E2E AIの内部モジュール検証ではなく、車両の閉ループ挙動として論証する（第8.4節参照）。

## 8.6 E2E AI出力に対する安全監視

E2E AIの出力は、無条件に車両へ通さない。ただし、安全監視の対象には意図的な制限を設ける。

### 8.6.1 安全監視系の役割原則

本書における安全監視系は、E2E AIが出した操舵・加速度指示の**場面に対する妥当性**（その場面で右に切るべきか左に切るべきか、加速すべきか減速すべきか、停止すべきか進行すべきか等）を判定する主体ではない。場面に対する妥当性の判断は、E2E AIそのものに委ねる（第8.1〜8.5節）。

安全監視系の役割は、E2E AIの出力が**場面理解を必要とせずに客観的に判定できる物理的・運動学的・連続性・整合性のエンベロープ**から外れていないかを監視することに限定する。すなわち、安全監視系は「AIの操舵指示が状況に対して正しいか」を判定するのではなく、「AIの操舵指示が車両として実行可能かつ物理的・運動学的に許容できる範囲に収まっているか」のみを判定する。

この設計上の選択は意図的である。理由は、E2E AIの場面妥当性を独立に判定する主体を置くと、その主体自体がE2E AIと同等の認識能力を持つ必要があり、安全論証が閉じなくなるためである（同じ前提を別のAIに置き換えただけになる）。本書はこの問題を回避するため、安全監視系を「**場面理解非依存のエンベロープ監視**」として位置付け、場面理解を要する妥当性は車両の閉ループ挙動論証（第8.4節）とAI Safety Case（第8.8節）で扱う。

### 8.6.2 監視対象

上記の役割原則に従い、安全監視系は以下を監視する。いずれの項目も、前方の障害物・歩行者・信号等の場面理解を必要とせずに判定できる項目に限定している。

| **監視対象** | **内容** | **場面理解の要否** |
| --- | --- | --- |
| 舵角指示 | 車速や機構上の物理的範囲に対して過大でないか（例：機構上の最大舵角の超過、車速に対して横加速度・ヨーレートが許容外となる急操舵） | 不要（車速と機構諸元のみで判定） |
| 出力変化 | 急変、振動、不連続がないか | 不要（連続フレーム間の出力差分のみで判定） |
| 推論周期 | 制御周期内に有効な出力が得られているか | 不要（時刻のみで判定） |
| 入力品質 | カメラ画像、時刻同期、車両状態量が利用可能か | 不要（信号レベルの健全性のみで判定） |
| 車両応答 | 指示に対して車両が物理的に期待どおり応答しているか | 不要（指示と実応答の整合のみで判定） |
| ODD状態 | ODD外または境界に近づいていないか | 不要（位置・地図・気象・通信品質等の客観量で判定） |

### 8.6.3 場面妥当性を監視対象としない理由

加速度指示および舵角指示の**場面妥当性**は、安全監視系の妥当性判定対象としない。

たとえば、ある場面で「急ブレーキを出すべきか否か」「右に避けるべきか左に避けるべきか」「停止線手前で止まるべきか進入すべきか」といった判断は、前方の障害物・歩行者・信号・他車・優先関係といった場面理解を要する。これを安全監視系で独立に判定するには、安全監視系自体がE2E AIと同等の認識・予測・判断能力を持たねばならず、論証として閉じない。

したがって、加速度指示の是非も、舵角指示の場面選択（どちらに切るべきか）の是非も、外部判定にはそぐわない。これらは、E2E AIによる出力と、結果としての車両の閉ループ挙動として論証する（第8.4節）。

加速度指示および舵角指示の値そのものについては、安全監視系が判定する物理的・運動学的エンベロープ監視（第8.6.2節）に加え、E2E protectionによるメッセージ完全性、車両側での物理範囲クリップ、車両応答の整合監視（VMC I/F仕様書6.2）で扱う。

### 8.6.4 検知時の挙動

安全監視系が、第8.6.2節のいずれかのエンベロープを逸脱したと判定した場合、通常走行を継続せず、MRC系ADSへフェイルオーバーしてMRCモードへ遷移する（第9章）。エンベロープ判定の定量閾値、ヒステリシス、判定の独立性確保は衛星文書「安全監視系設計書」で確定する。

## 8.7 E2E AIの限界とMRC遷移

E2E AIは通常走行の中核であるが、MRC成立の唯一の根拠ではない。以下の場合には、E2E AIの通常出力に依存せず、MRCモードへ遷移する。

- 入力画像が必要な品質を満たさない
- 推論が停止または遅延する
- 出力が安全監視系のエンベロープを逸脱する（第8.6.2節の物理的・運動学的・連続性・整合性のエンベロープ。場面妥当性ではなく客観的に判定可能な範囲を逸脱する場合）
- ODDを逸脱する
- センサ、計算機、通信、電源、制動、操舵に失陥がある
- 通常走行ADSが停止または異常出力を行う

MRC遷移の主体、検知から遷移完了までの時間制約、フェイルオーバー時の二重指令防止・無指令状態防止は第9章で扱う。MRC中の車両挙動は第11章で扱う。

# 9. 多重系ADSによる失陥時安全

## 9.1 多重系ADSを採用する目的

Turingは、L4車両の失陥時安全（サブゴール(b)、第4.5節）を成立させるため、ADSを多重系として構成する。

多重系ADSの目的は、一系統が失陥した後に通常走行を継続することではない。一系統が失陥しても、MRC系ADSがMRCモードへ遷移し、車両をリスク最小状態へ移行させることである。すなわち、Turingの多重系ADSは「Fail-Operationalで通常走行を続ける構成」ではなく、「Fail-DegradedでMRCを成立させる構成」として位置付ける。

第2版では、本章はISO 26262（多重系の故障と多重系のCCF・Cascading Failure・Latent Faultの管理）、ISO/TS 5083（多重系を含むADS全体の安全論証骨格）、UN-R 157 5.1.5 MRMの考え方と整合させる。形式的準拠は目的としない（第6.1節）が、対外説明・協議の共通言語として上記規格群を用いる。

## 9.2 通常走行ADSの役割

通常走行ADSは、通常走行時に主としてE2E AI（第8章）を実行し、舵角指示および加速度指示を生成する。

通常走行ADSは、ODD内で通常走行が可能であり、入力品質、出力、車両応答が安全監視系（第9.5節）により健全と判断される場合に、通常走行モードを継続する。通常走行ADSの計算機実装は通常走行用ECU、入力は通常走行用カメラ群7台（第10.2節）とする。

## 9.3 MRC系ADSの役割

MRC系ADSは、通常走行ADSに失陥が生じた場合に、自動的に起動または制御権限を引き継ぎ、MRCモードへ遷移する。

MRC系ADSは、通常走行を完全に代替することを主目的としない。MRC系ADSの役割は、能力の制限ではなく、行動範囲の限定として整理する。すなわち、MRC系ADSはMRC専用カメラ1台（前方広角）を入力とし、ODD内の動的環境（歩行者、他車両、信号等）への衝突回避能力を持ったうえで、行動目的を「同一車線内に安全に停止する」ことに限定する。

MRC系ADSの目的は、以下である。

1. 通常走行ADSの失陥時に無指令状態を避ける
2. 危険な通常走行ADS出力を遮断する
3. MRC専用カメラを用いて、同一車線内の衝突回避と安全停止を実行する
4. 車両を減速または停止させる
5. MRC完了後に再発進を制限する

MRC系ADSの計算機実装はMRC用ECU、入力はMRC専用カメラ1台と本書で定める。詳細な異種ECU冗長アーキテクチャは第9.4節で、実装方針は第9.11節で扱う。

## 9.4 異種ECU冗長アーキテクチャ

第2版では、多重系ADSの中核として**異種ECU冗長アーキテクチャ**（Heterogeneous ECU Redundancy）を採用する旨を独立節として明示する。具体的には、通常走行ADSの実装に通常走行用ECU、MRC系ADSの実装にMRC用ECUを用い、両ECUは異なる世代・異なる物理設計・異なる製造マスクのシリコンを持つ異種ECUとする。

| **層** | **通常走行ADS** | **MRC系ADS** | **分離の意味** |
| --- | --- | --- | --- |
| シリコン | 通常走行用ECU（高性能AI処理向けSoC） | MRC用ECU（MRCタスク十分性能） | 同一シリコンの物理欠陥CCFを設計排除 |
| ファームウェア / OS | 通常走行用ECU FW、OS | MRC用ECU FW、OS | 同一FW/OSバグの同時顕在化を設計排除 |
| ツールチェーン | 通常走行用E2E AIの学習・コンパイル系 | MRC専用E2E AIの学習・コンパイル系 | 同一ツールチェーン由来欠陥の同時顕在化を設計排除 |
| AIモデル | 通常走行用E2E AI（広汎な走行場面） | MRC専用E2E AI（衝突回避・車線内停止に特化） | 同一モデルの予測バイアス由来CCFを設計排除 |
| 入力センサ | 通常走行用カメラ群7台 | MRC専用カメラ1台（前方広角、MRC用ECU直結） | 共通センサ入力CCFを設計排除（第10.2節） |
| 電源 | LV1主給電 | LV2主給電（独立） | 単一電源故障による両系同時喪失を設計対策（第10.9節） |
| 開発チーム | 通常走行ADSチーム | MRC系ADSチーム | 同一systematic failure（同一設計過誤）を設計排除 |

異種ECU冗長アーキテクチャにより、通常走行ADSとMRC系ADSの間のCCFは、シリコン・ファームウェア・OS・ツールチェーン・AIモデル・入力センサ・開発チームの各層で設計上排除される。残存する共通故障源（共通電源、共通通信、共通環境、共通サプライヤ、共通製造ロット、共通OTA等）については、第9.10節および第10章で物理的・設計的・運用的に対処する。

異種ECU冗長アーキテクチャの選択理由、各ECUの仕様、両ECU間のArbitration、I/F、検証手順、量産プロセスは、衛星文書「多重系ADS設計書」「異種ECU冗長アーキテクチャ仕様書（草稿：`05_Chapter9_HeterogeneousSoC_draft.md`）」「車両要求仕様書」で具体化する。

## 9.5 安全監視系の役割

安全監視系は、通常走行ADSの状態、入出力、車両応答、ODD適合性、システム健全性を継続的に監視し、異常検知時にMRC系ADSへのフェイルオーバーまたはMRCモードへの遷移を要求する。

安全監視系の検知対象は、以下である。

| 検知カテゴリ | 検知対象の例 |
| --- | --- |
| 推論異常 | 通常走行ADSの停止、ハング、推論遅延 |
| 出力異常 | 異常な舵角指示（物理範囲超過、車速に対する過大）、出力の急変・振動・不連続 |
| 入力品質 | 入力画像の異常（フリーズ、欠落、汚損、白飛び・黒潰れ）、時刻同期異常 |
| 車両応答 | 指示に対する車両応答の不整合 |
| ODD適合性 | ODD逸脱、ODD境界接近、Triggering Conditionの顕在化（第7.4節） |
| 通信・時刻 | 安全関連通信断、遅延、時刻同期ずれ |
| 電源 | 電源系異常、LV1/LV2残存容量低下 |
| AI不確実性 | E2E AIが申告する高不確実性、Out-of-Distribution検知 |

安全監視系は通常走行ADSと独立に動作することを前提とし、独立電源・独立通信経路・独立計算リソースで実装する。具体的な実装上の独立性、判定ロジック、ヒステリシス、ログ記録は衛星文書「安全監視系設計書」で扱う。

## 9.6 一系統失陥時の自動フェイルオーバー

通常走行ADSが失陥した場合、MRC系ADSは自動的に起動または制御権限を引き継ぐ。

フェイルオーバー時には、以下を防止する必要がある。

- 通常走行ADSとMRC系ADSからの二重指令
- どちらの系統からも指令が出ない無指令状態
- 危険な通常走行ADS出力の継続
- フェイルオーバー後の通常走行継続
- MRC完了後の自動再発進

二重指令防止・無指令状態防止のためのArbitration（調停）は、車両側で独立に実装する。すなわち、Arbitration層を通常走行ADSとMRC系ADSのいずれにも依存させない構成とする。Arbitrationの実装、Plausibility Check、物理範囲クリップ、車両応答整合監視は衛星文書「VMC I/F仕様書 6.2」および「車両要求仕様書」で具体化する。

## 9.7 フェイルオーバー後のMRCモード遷移

フェイルオーバー後、MRC系ADSは通常走行モードではなくMRCモードへ遷移する。

この方針により、一系統失陥後のリスクを、通常走行継続ではなく、安全停止またはリスク最小状態への移行として扱う。MRCモードでの車両挙動、行動制限、停止位置選定の考え方は第11章で扱う。

## 9.8 通常走行への自動復帰禁止

フェイルオーバー後は、通常走行への自動復帰を禁止する。

理由は、失陥原因が未確定の状態で通常走行へ復帰すると、同一故障の再発、別系統への影響、共通原因故障の見落としにより、リスクが増大する可能性があるためである。

MRC完了後は、車両を停止状態に保持し、遠隔監視または現場対応を待つ（第11.7節、第12章）。再発進の判断主体、判断条件、再発進前の確認項目は衛星文書「運用手順書」「特定自動運行計画書」で扱う。

## 9.9 両系統異常時の扱い

通常走行ADSとMRC系ADSの両方に異常がある場合、車両は可能な範囲でより保守的な緊急停止へ移行する。

この場合、車両は通常走行判断を行わず、直近の安全確認結果、車両状態量、制動・操舵の残存能力に基づき、リスクを最小化する挙動を取る。具体的には、駆動遮断、制動優先、ハザード点灯、外向けHMI、open-loopでの最大限保守的な減速を組み合わせる（第10章、第11.8節）。

両系統異常は本書の安全論証では「設計上発生しないとは主張できない残存事象」として扱う。両系統異常の発生確率を許容水準まで低減するための、CCF設計排除・設計対策・運用対策は第9.10節で扱う。

## 9.10 共通原因故障の考え方

多重系ADSでは、単に系統を二重化するだけでは不十分である。本節では、ISO 26262が扱う多重系の以下の三つの観点を統合的に扱う。

1. **Common Cause Failure（CCF, 共通原因故障）**：同じ原因で両系統が同時に失われる事象
2. **Cascading Failure（カスケード故障）**：一系統の故障が他系統へ波及する事象
3. **Latent Fault（潜在故障）**：直ちに機能喪失を起こさないが、別の故障と組み合わさって安全機能を喪失させる潜在的故障

### 9.10.1 CCFの分類と本書での対応

CCFを設計上の対応によって以下に分類する。「設計排除」は第9.4節の異種ECU冗長アーキテクチャにより設計上発生しないことを主張する。「設計対策」は物理分離・冗長化により発生確率を許容水準まで低減することを主張する。「運用対策」は運用後監視・運用手順により管理することを主張する。「残存」は許容残存リスクとして第21章（第2版TOC）で扱う。

| **CCFカテゴリ** | **例** | **本書での対応** | **対応の根拠節** |
| --- | --- | --- | --- |
| 同一シリコン由来 | 同型ECUの物理欠陥 | 異種ECU冗長で**設計排除** | 9.4 |
| 同一ファームウェア由来 | 同一FWのバグ | 異種ECU冗長で**設計排除** | 9.4 |
| 同一OS由来 | 同一OSの脆弱性・バグ | 異種ECU冗長で**設計排除** | 9.4 |
| 同一ツールチェーン由来 | 同一コンパイラのバグ | 異種ECU冗長で**設計排除** | 9.4 |
| 同一AIモデル由来 | 同一モデルの予測バイアス | 異種ECU冗長で**設計排除** | 9.4 |
| 同一systematic failure | 同一開発チームの設計過誤 | 異種ECU冗長（チーム分離）で**設計排除** | 9.4 |
| 同一センサ入力 | 共通カメラの故障 | MRC専用カメラを通常走行用カメラ群と独立配置することで**設計排除** | 第10.2節 |
| 共通電源 | LV1喪失で両系停止 | LV1/LV2分離で**設計対策** | 第10.9節、車両要求仕様書 |
| 共通通信経路 | バックボーン1系統喪失 | Dual ring + FRERで**設計対策** | 第10.10節、VMC I/F仕様書 |
| 共通時刻同期 | PTP Grandmaster失陥 | PTP冗長で**設計対策** | 第10.10節 |
| 共通環境（熱・振動・水・電磁ノイズ）| 同一筐体内の同時被災 | 物理的セパレーションで**設計対策** | 車両要求仕様書 |
| 共通環境曝露（雨・霧・夜間・逆光）| フロントガラス越し共通曝露 | **残存リスク** | 第21章（第2版TOC） |
| 共通サプライヤ脆弱性 | ECUサプライヤ共通CVE | 運用後監視で**運用対策** | Cybersecurity Case |
| 共通製造ロット欠陥 | 同一ロット由来不良 | 製造ロット分散発注で**運用対策** | 車両要求仕様書 |
| 共通OTA更新不具合 | 不正な同時更新 | rolling update + ロールバックで**運用対策** | 第6.7節、モデル更新管理書 |

### 9.10.2 Cascading Failureの考え方

Cascading Failureとして、以下の波及パスを考慮する。

| 波及パス | 例 | 対処 |
| --- | --- | --- |
| 制御指令経由 | 通常走行ADSの異常出力が車両制御I/Fを介してMRC系ADSの判断に影響 | Arbitrationの車両側独立実装、Plausibility Check、物理範囲クリップ |
| Arbitration経路 | Arbitration層自体の誤動作で両系統指令が同時に通る | Arbitrationの独立実装、安全監視系の独立判定 |
| 物理的近接 | 熱・振動・電磁ノイズの波及 | 物理的セパレーション、独立筐体、独立配線経路 |
| 共通通信媒体 | 通常走行ADSが共通通信経路をフラッディング | ネットワーク分離、QoS、レート制限 |
| 共通電源負荷 | 通常走行ADSの異常電流が共通電源系を巻き込む | LV1/LV2分離、電源側保護回路 |

### 9.10.3 Latent Faultの考え方

Latent Faultは、それ単独では機能喪失を起こさないが、別の故障と組み合わさって安全機能を喪失させる潜在的故障である。本書では以下の枠組みでLatent Faultを管理する。

| 対象 | Latent Faultの例 | 検知手段 |
| --- | --- | --- |
| MRC系ADS | 通常時には起動していない期間のMRC専用E2E AIまたはMRC用ECUのSilent Failure | 定期的な自己診断、ハートビート、Built-In Self-Test、シャドウ実行 |
| 安全監視系 | 安全監視系自身が異常時に検知できない事象 | 安全監視系の独立Watchdog、定期診断 |
| MRC専用カメラ | 通常時に使用しないMRC専用カメラのSilent Failure | カメラ健全性常時監視（第10.5節） |
| 冗長電源（LV2） | LV1健全時にLV2の劣化が見えない | LV2の常時監視、定期負荷試験 |
| 冗長通信経路 | 主経路健全時に副経路の障害が見えない | 副経路の常時ヘルスチェック、フェイルオーバー試験 |

ISO 26262 Part 5のLFM（Latent Fault Metric）の定量目標と、それを達成するための診断カバレッジの設計は、衛星文書「車両要求仕様書」「安全監視系設計書」「各ECU Safety Manual」で確定する。本書本体ではこれらを観点参照として位置付ける。

### 9.10.4 CCF/Cascading/Latent Fault対応の安全主張

第9.4節の異種ECU冗長アーキテクチャと第9.10.1〜9.10.3の対策により、本章のCCF/Cascading/Latent Fault対応の安全主張は以下である。

> 通常走行ADSとMRC系ADSの同時機能喪失は、シリコン・ファームウェア・OS・ツールチェーン・AIモデル・systematic failure・センサ入力に起因しては設計上発生しない。共通電源・通信・時刻同期・環境・サプライヤ・ロット・OTAに起因するCCFと、制御指令・Arbitration・物理近接を経由するCascading Failureについては、車両アーキテクチャの分離設計と運用後監視により、許容残存リスク水準まで低減する。Latent Faultについては、定期診断、独立Watchdog、自己診断、シャドウ実行により、組合せ故障による安全機能喪失を許容残存リスク水準まで低減する。共通環境曝露（雨・霧・夜間・逆光等のフロントガラス越し共通入力劣化）は許容残存リスクとして管理する。

## 9.11 MRC系ADS実装方針

MRC系ADS実装方針は、第9.4節の異種ECU冗長アーキテクチャを採用方針として確定する。基本方針は以下である。

- MRC系ADSの目的は通常走行の継続ではなく、同一車線内安全停止というMRC達成であること
- MRC系ADSは通常走行ADSとは異種ECU冗長アーキテクチャ（第9.4節）で構成されること
- フェイルオーバー後は通常走行へ自動復帰しないこと（第9.8節）
- MRC系ADSは通常走行ADSとの二重指令および無指令状態を発生させないこと（第9.6節）
- MRC系ADSはCCFを考慮した物理的・設計的分離を前提とすること（第9.10節）
- MRC完了後の停止保持、ハザード、外向けHMI、ログ記録を確実に行うこと（第11章、第13章）
- MRC専用E2E AIに対するAI Safety Requirements導出は通常走行用E2E AIと並行して行うこと（第8.8節、AI Safety Case）

具体的なECU個別設計、I/F、Arbitration実装、MRC専用E2E AIの仕様、検証手順、量産プロセスは、衛星文書（多重系ADS設計書、車両要求仕様書、AI Safety Case、MRC設計書、異種ECU冗長アーキテクチャ仕様書）で具体化する。

# 10. センサ・計算機・車両制御系の失陥安全

## 10.1 ハードウェア失陥をE2E AIで吸収しない理由

Turingの安全論証では、ハードウェア失陥をE2E AIの通常機能で吸収する対象とはしない。

カメラフリーズ、SoCハング、通信断、電源低下、制動系異常、操舵系異常などは、E2E AIの性能問題ではなく、車両アーキテクチャとして扱うべき失陥である。第8章のE2E AI安全論証はAIの性能限界（ISO/PAS 8800、SOTIF語彙借用）を扱うのに対し、本章は機能安全（ISO 26262）の枠組みでハードウェア失陥を扱う。両者を混同しない。

第2版では、本章の構造を以下の3層縮退構造に再構成して扱う。

| **層** | **対象** | **縮退の主体** | **本章の節** |
| --- | --- | --- | --- |
| 入力層 | カメラ（通常走行用7台、MRC専用1台） | 安全監視系＋通常走行用E2E AI＋MRC系ADS | 第10.2〜10.5節 |
| 計算層 | 通常走行用ECU、MRC用ECU、安全監視系 | 異種ECU冗長＋安全監視系 | 第10.6節 |
| 制御・基盤層 | 制動、操舵、駆動、電源、通信、時刻同期 | 車両側冗長＋縮退制御 | 第10.7〜10.10節 |

各失陥に対する定量要求（残存制動能力、減速度、応答時間、電源容量、通信遅延等）は、本書では確定しない。これらは衛星文書「車両要求仕様書」「VMC I/F仕様書」「各ECU Safety Manual」で確定する。本章では各失陥に対する基本挙動と縮退方針を示す。

## 10.2 カメラ構成と独立性

本書が前提とするカメラ構成は、以下である。

| **構成** | **対象ADS** | **役割** |
| --- | --- | --- |
| 通常走行用カメラ群（7台） | 通常走行ADS | 通常走行用E2E AIの入力。フロント広角、フロントワイド、左右前方、左右後方、後方の7台で広視野・冗長な認識を提供する |
| MRC専用カメラ（1台） | MRC系ADS | MRC専用E2E AIの入力。前方広角1台。MRC用ECUに直結 |

通常走行用カメラ群とMRC専用カメラは、物理・電気・伝送系で完全独立とする。具体的には、両系統は独立したECU、独立した電源系統（LV1/LV2）、独立したケーブル経路を持つ。これにより、カメラ層のCCFリスクを設計上排除する（第9.10節「同一センサ入力」CCFカテゴリ）。

各カメラの解像度、HDR、視野角、フレームレート、ISP仕様、取付位置、キャリブレーション要件は、衛星文書「車両要求仕様書」で確定する。

## 10.3 通常走行用カメラ群の縮退能力

通常走行ADSは、7台のカメラのうち1台が失陥しても、残存6台のカメラを用いて、ODD内で安全な停止位置までの走行と停止を実行する能力を持つ。これは通常走行用E2E AIの設計要件であり、AI Safety Requirements（第8.8節）の性能要件カテゴリで具体化する。

7台のうち1台失陥時の挙動は、以下を基本とする。

- 失陥カメラの役割と残存6台のカバー範囲を安全監視系が判定する
- 通常走行の継続条件を満たさない場合、計画的に通常走行を終了し、安全停止位置に停止する
- 通常走行の継続条件を満たすが不確実性が高い場合、MRCモードへの遷移を許容する
- 2台目以降のカメラ失陥が発生した場合、または前方認識に重大な影響がある単独失陥の場合、即座にMRC系ADSへフェイルオーバーしてMRCモードへ遷移する

通常走行用カメラ群の単一カメラ失陥は、MRC系ADSの能力に影響しない。MRC系ADSはMRC専用カメラを独立に持つため（第10.2節、第9.4節）、入力センサ層でのCascading Failureは設計上排除される。

各カメラ失陥時の継続可否判定の定量基準（残存FOV、前方認識の十分性、推論時間制約等）は、衛星文書「AI Safety Case」「安全監視系設計書」で確定する。

## 10.4 MRC専用カメラ失陥時の基本挙動

MRC専用カメラに異常が発生した場合、MRC系ADSは入力を喪失し、通常のMRC実行能力を失う。

この場合の対処は、以下である。

- 通常走行ADSと通常走行用カメラ群が健全であれば、通常走行ADSは通常走行を継続するが、MRC capability喪失状態を申告し、計画的に通常走行を終了して安全停止位置に停止する
- 通常走行ADSも同時に失陥している場合、両系統異常時の保守的緊急停止（第9.9節）に移行する。MRC系ADSは直近の車両状態量と制御残存能力に基づき、open-loopでの最大限保守的な減速・停止を実行する

MRC capability喪失状態の申告主体、申告先、申告タイミング、計画的通常走行終了の経路選定は、衛星文書「MRC設計書」「VMC I/F仕様書」で確定する。

## 10.5 カメラ健全性の考え方

カメラが映像を出していることと、通常走行またはMRCに使えることは異なる。本書では、健全性を通常走行用カメラ群とMRC専用カメラそれぞれに対して定義する。

カメラ健全性の共通項目は、以下である。

| **項目** | **内容** |
| --- | --- |
| フレーム連続性 | 欠落、フリーズ、重複、異常遅延がない |
| 露出・HDR | 白飛び、黒潰れ、逆光に対して必要な情報が残る |
| 汚損・曇り | 水滴、泥、雪、曇り、虫付着等を検知できる |
| 同期 | 車両状態量および制御周期と整合している |
| キャリブレーション | 取付ずれ、振動ずれが許容範囲内である |
| 通信完全性 | 画像破損、伝送欠落、タイムスタンプ異常を検知できる |
| 熱・電圧 | カメラ、ISP、伝送系が動作保証範囲内にある |

通常走行用カメラ群については、上記に加え、複数カメラ間の整合性（オーバーラップFOVでの認識整合）を健全性に含める。

MRC専用カメラについては、本カメラがMRC系ADSの唯一の入力であるため、上記の健全性項目を確実に監視する。常時運用中（通常走行モードでもMRC専用カメラは健全性監視対象）であることにより、MRC専用カメラのLatent Fault（第9.10.3節）を検知する。健全性低下を検知した場合、安全監視系はMRC capability低下を申告し、車両は第10.4節の挙動に従う。

カメラ健全性の定量閾値、判定アルゴリズム、ヒステリシス、診断カバレッジは、衛星文書「カメラ健全性定義書」「安全監視系設計書」で確定する。

## 10.6 ECU失陥時の基本挙動

第9.4節の異種ECU冗長アーキテクチャにより、通常走行用ECUとMRC用ECUは異なるシリコン・FW・OS・ツールチェーン・AIモデルで構成される。これにより、ECU層の同一原因CCFは設計上排除される（第9.10.1節）。

通常走行用ECUまたはMRC用ECUに異常が発生した場合、安全監視系は以下を検知する。

| 検知対象 | 内容 |
| --- | --- |
| 推論停止 | 推論プロセスが応答しない |
| 推論遅延 | 制御周期内に出力が得られない |
| ハング | OSまたはアプリケーションが応答しない |
| プロセス異常 | 主要プロセスのクラッシュまたは再起動 |
| メモリ異常 | ECCエラー、メモリ枯渇、メモリ破損 |
| 異常出力 | 物理範囲外、急変、不連続な出力 |
| 通信途絶 | ハートビート喪失、通信品質劣化 |
| 自己診断異常 | Built-In Self-Test、Watchdog、温度・電圧監視の異常 |

異種ECU冗長を反映した基本挙動は以下である。

- 通常走行用ECUの失陥：MRC用ECUが起動するMRC系ADSがMRCモードへ遷移（第9.6〜9.7節）。通常走行用ECUの異常出力は、車両側Arbitration（第9.6節）と安全監視系により遮断する。
- MRC用ECUの失陥：MRC capability喪失を申告し、通常走行ADSが計画的に通常走行を終了（第10.4節と同じ縮退経路）。MRC用ECUのLatent Faultは定期的な自己診断・シャドウ実行で検知する（第9.10.3節）。
- 安全監視系の失陥：安全監視系自身の独立Watchdogにより検知し、検知不能事象（Latent Fault）を最小化する。安全監視系の独立性は衛星文書「安全監視系設計書」で確定する。

ECU失陥検知の定量応答時間、診断カバレッジ（SPFM、LFM）、フェイルオーバー時間目標は、衛星文書「車両要求仕様書」「各ECU Safety Manual」で確定する。本書本体はISO 26262 Part 5の指標を観点参照する位置付けとする（第6.3節）。

## 10.7 制動系失陥時の基本挙動

制動系に異常が発生した場合でも、可能な限り残存する制動能力を用いてMRCを達成する。

制動系に求める基本方針は以下である。

- 主制動系の異常を検知する
- 残存制動能力を把握する
- 駆動遮断を優先する
- 制動優先を確保する（駆動指令と制動指令が競合する場合は制動優先）
- 減速度を安全側に制御する
- 停止後に車両を保持する（パーキング、電動パーキングブレーキ等）

主制動系・冗長制動系の構成、残存制動能力の検知方法、減速度の安全側制御、停止後保持の具体仕様、制動応答時間、フェイルセーフ動作は、衛星文書「車両要求仕様書」「VMC I/F仕様書」で確定する。本書本体ではUN-R 157 5.1.1（衝突回避）および5.1.5（MRM）の観点を参照する。

## 10.8 操舵系失陥時の基本挙動

操舵系に異常が発生した場合、車両は可能な範囲で現在車線を維持しながら減速・停止する。

操舵が制限される場合、積極的な回避、車線変更、路肩退避を前提にせず、制動主体のMRCを基本とする（第11.4節）。

操舵主系・冗長系の構成、操舵異常時の権限制限、残存操舵能力の判定、操舵失陥モード別の挙動、操舵応答時間は、衛星文書「車両要求仕様書」「VMC I/F仕様書」で確定する。

## 10.9 電源失陥時の基本挙動

電源失陥時には、MRC関連機能を優先する。優先すべき機能は以下である。

- MRC系ADS
- 安全監視系
- MRC制御機能
- 制動
- 操舵
- 駆動遮断
- ハザード
- 外向けHMI
- ログ記録

単一電源故障により、MRC関連機能が同時に喪失しない構成を目指す。具体的には、LV1（通常走行ADS主給電）とLV2（MRC系ADS主給電）を分離し、LV1喪失時にもLV2配下のMRC機能が動作することを設計対策とする（第9.10.1節「共通電源」CCFカテゴリ）。

LV1/LV2の容量、保持時間、自動切替条件、低電圧時の優先度、両電源同時喪失時の縮退挙動、ECU別の電源系統割付は、衛星文書「車両要求仕様書」で確定する。

## 10.10 通信・時刻同期失陥時の基本挙動

通信断、通信遅延、時刻同期ずれが発生した場合、車両は入力・出力の信頼性を安全側に評価する。

通信・時刻同期に異常がある場合には、通常走行継続を避け、MRCモードへ遷移する。具体的には、車両内バックボーン通信のDual ring + FRER構成により、単一経路喪失時に通信が継続することを設計対策とする（第9.10.1節「共通通信経路」CCFカテゴリ）。時刻同期はPTP冗長構成により、PTP Grandmaster失陥時にも時刻同期が維持されることを設計対策とする（同「共通時刻同期」CCFカテゴリ）。

遠隔監視との通信断は、MRC成立の妨げとしない。すなわち、車両単独でMRCを完了する設計とする（第12.2節）。

通信品質の定量基準（許容遅延、許容ロス率、許容ジッタ）、時刻同期精度、Dual ring/FRER/PTP冗長の実装、安全関連通信のメッセージ仕様は、衛星文書「VMC I/F仕様書」「車両要求仕様書」で確定する。

# 11. MRC設計の基本思想

## 11.1 MRCの目的

MRC（Minimal Risk Condition）の目的は、故障、機能低下、ODD逸脱、E2E AI出力異常等が発生した場合に、車両をリスク最小状態へ移行させることである。

TuringのL4自動運転車両におけるMRCは、**MRC系ADSが、MRC専用カメラ1台を入力として、衝突を回避しながら同一車線内に安全に停止すること**を中核として明示する。MRCは目的地への走行継続ではなく、複雑な経路選択、車線変更、追越し、合流、回避操作を行わず、行動目的を「同一車線内に止まる」ことに絞る。

MRCは、事故リスクを下げるために、車両の行動を制限し、安全側へ遷移する挙動である。MRCの実行可能性（MRC capability）は、MRC系ADS、MRC用ECU、MRC専用カメラ、車両側の制動・操舵・電源・通信が健全であることに依存する。これらの一部または全部が失陥した場合の挙動は、第10章および第9.9節（両系統異常時の扱い）で扱う。

第2版では、MRCをUN-R 157 5.1.5のMinimum Risk Manoeuvre（MRM）の概念と整合させて扱う（第6.10節）。形式的準拠は目的としない（第6.1節）が、対外説明・協議の共通言語としてMRMの概念を用いる。

## 11.2 MRC開始条件の考え方

MRCは、以下の場合に開始する。

- 通常走行ADSが失陥した場合
- MRC系ADSへのフェイルオーバーが発生した場合
- E2E AI出力が安全制約に反する場合
- 入力画像が必要な品質を満たさない場合
- ODDを逸脱した場合
- カメラ、SoC、ECU、通信、電源、制動、操舵に失陥がある場合
- 遠隔監視断ではないが、他の安全条件により通常走行が不適切と判断される場合

MRC開始判定の主体は安全監視系（第9.5節）または通常走行ADS自身（自己申告）である。MRC開始から完了までの時間制約、判定アルゴリズム、ヒステリシスは、衛星文書「MRC設計書」「安全監視系設計書」で確定する。

## 11.3 MRCモードで制限する行動

MRCモードでは、以下の行動を制限する。

| **行動** | **MRC時の扱い** |
| --- | --- |
| 車線変更 | 原則実施しない |
| 追越し | 実施しない |
| 後退 | 実施しない |
| 通常右左折 | 原則実施しない |
| 合流 | 原則実施しない |
| 交差点進入 | 不確実な場合は停止する（第11.5節） |
| 横断歩道通過 | 横断者不在が不確実な場合は停止する（第11.5節） |
| 路肩退避 | 側方・後方確認が成立する場合に限定する |
| 現在車線内停止 | デフォルト動作とする（第11.4節） |

行動制限の具体的な判定基準、優先度、競合時の扱いは、衛星文書「MRC設計書」で確定する。

## 11.4 現在車線内停止の考え方

### 11.4.1 デフォルトとしての現在車線内停止

MRCでは、現在車線内停止をデフォルト動作とする。

これは、失陥時に側方・後方確認が不十分なまま車線変更や路肩退避を行うことが、かえってリスクを増大させる可能性があるためである。MRC系ADSの入力はMRC専用カメラ1台（前方広角）に限定されており、側方・後方の認識能力は通常走行用カメラ群に比べて限定的であることから、現在車線内停止を基本選択とする。

### 11.4.2 「停止することがかえって危険な場面」の識別

ただし、現在車線内停止が常に最良の選択とは限らない。第2版では、停止より走行継続が安全な場面、すなわち「停止すること自体が周囲リスクを増大させる場面」を識別し、別の挙動を選択する考え方を明示する。

具体的には、以下の場面では、現在車線内即時停止ではなく、可能な範囲で短距離の前進を伴う停止位置調整を許容する。

| 場面カテゴリ | 例 | 即時停止が増大させ得るリスク | 望ましいMRC挙動の例 |
| --- | --- | --- | --- |
| 交差点内 | 交差点中央、横断歩道上、停止線手前直近 | 直進・右左折交通の阻害、横断者との接触 | 交差点を抜けた直後の車線内位置で停止 |
| 横断歩道直上 | 信号横断歩道、無信号横断歩道 | 横断者の通行妨害、視界遮蔽 | 横断歩道を抜けた直後の車線内位置で停止 |
| 鉄道踏切 | 踏切内 | 列車との衝突 | 踏切を抜けた直後の車線内位置で停止 |
| カーブ・見通し不良 | 急カーブ内、丘の頂上 | 後続車からの視認性不足、追突 | 直線路または視認性確保位置で停止 |
| トンネル内 | 短中尺トンネル | 後続車視認性不足、避難困難 | トンネル出口直後で停止（出口が近い場合） |
| 自動車専用道合流部 | 合流車線、加速車線 | 合流車との接触 | 本線車線または安全域への移動 |
| バス停・タクシー乗降場 | 公共交通停車域 | 公共交通の運行妨害 | 公共交通帯外への短距離移動 |
| 緊急車両通行帯 | 消防・救急通行帯 | 緊急車両通行妨害 | 緊急車両通行帯外への短距離移動 |

### 11.4.3 MRC位置選定アルゴリズムの考え方

MRC位置選定は、即時停止と短距離前進停止のいずれが周囲リスクを最小化するかを判定する考え方として整理する。本書では位置選定アルゴリズムの詳細は確定しないが、考え方の枠組みを以下に示す。詳細は衛星文書「MRC設計書」で確定する。

判定の入力は以下である。

- MRC系ADSの認識結果（前方広角カメラ1台による前方車線・障害物・横断者の状況）
- 安全監視系が把握する車両状態（残存制動能力、残存操舵能力、車速、車線情報）
- 地図情報（交差点・横断歩道・踏切・トンネル等の特殊地理特徴の近接性）
- 通常走行ADS失陥前の最終認識結果（フェイルオーバー直前の周囲状況スナップショット）

判定の出力は以下のいずれかである。

| 選択肢 | 適用場面 |
| --- | --- |
| 即時停止 | 通常の走行環境、特殊地理特徴から離れた位置 |
| 短距離前進停止 | 第11.4.2節の「停止すべきでない場面」に車両が位置している場合 |
| open-loop保守的減速 | MRC系ADSも失陥した両系統異常時（第9.9節）。MRC位置選定は機能しない |

判定の制約は以下である。

- 短距離前進停止は、MRC系ADSが衝突回避能力を持つ範囲内でのみ選択する
- 短距離前進停止は、追加移動距離・追加時間が許容上限を超えない範囲とする
- 短距離前進停止により、より大きな新たなリスク（ODD逸脱、追加交差点進入等）を発生させない
- 判定にあたり通信・遠隔監視には依存しない（車両単独で判定する、第12.2節）

判定の許容上限値（追加移動距離、追加時間、車速制約等）、特殊地理特徴の地図側付与方法、判定時間予算、フェイルバック挙動は、衛星文書「MRC設計書」で確定する。

### 11.4.4 路肩退避の扱い

路肩退避は、側方・後方確認が成立し、周囲交通へのリスクが現在車線内停止よりも低いと判断できる場合に限定する。Phase 1（実証フェーズ）では、MRC系ADSの入力がMRC専用カメラ1台に限定されることから、路肩退避は基本選択肢としない。Phase 2以降での扱いはAppendix「段階的社会実装ロードマップ」（第2版TOC第19章）で扱う。

## 11.5 交差点・横断歩道付近での考え方

交差点や横断歩道付近では、不確実性を安全側に扱う。

信号、停止線、横断者、前方空間、側方交通が不確実な場合には、進入を抑制し、停止を優先する。

第2版では、第11.4.2節と整合させ、交差点・横断歩道付近のMRC挙動を以下に整理する。

| 状況 | MRC挙動の基本方針 |
| --- | --- |
| 交差点進入直前（停止線手前にまだ余裕がある） | 停止線手前で停止 |
| 交差点進入中（停止線通過済、交差点中央付近） | 第11.4.2節に従い、交差点を抜けた直後の車線内位置で停止 |
| 横断歩道接近中 | 横断歩道手前で停止 |
| 横断歩道直上 | 第11.4.2節に従い、横断歩道を抜けた直後の車線内位置で停止 |
| 交差点・横断歩道双方の近接 | 双方を抜けた位置を優先選択 |
| 交差点内で停止が不可避 | 交差点中央付近より、進路前方寄りの位置を選択（後続車との衝突リスク低減） |

判定主体、判定タイミング、優先度競合時の扱い、信号現示変化中の扱いは、衛星文書「MRC設計書」で具体化する。

## 11.6 ハザード・外向けHMI

MRC中および停止後には、周囲交通に対して車両状態を知らせる必要がある。

そのため、ハザードまたは外向けHMIを用いて、車両が通常走行状態ではないことを周囲に示す。ハザードの点灯条件、外向けHMIの表示内容（無人運転中、停止中、緊急停止中、回収待機中等の区別）、視認距離、夜間視認性は、衛星文書「車両要求仕様書」「外向けHMI仕様書」で確定する。

## 11.7 停止後の状態保持

MRC完了後、車両は停止状態を保持する。

- 停止後は、自動再発進を行わない（第9.8節）
- パーキングまたは電動パーキングブレーキにより車両を物理的に保持する
- ハザード・外向けHMIを継続点灯する
- 走行ログを継続記録する（第13章）
- 必要に応じて、遠隔監視、現場対応、手動回収、乗客対応に移行する（第12章）

停止保持の具体的な保持手段、保持時間、低電力モードへの遷移、再起動手順は、衛星文書「MRC設計書」「車両要求仕様書」で確定する。

## 11.8 MRC失敗時の考え方

MRCが計画どおりに完了できない場合（両系統異常時、複合的な制動・操舵失陥、open-loop制御に縮退した場合等）でも、車両は可能な範囲でリスクを低減する。

例として、以下を行う。

- 駆動遮断
- 制動優先
- ハザード点灯
- 外向けHMI表示
- 車両状態の記録（第13章）
- 遠隔監視への通知（通信が利用可能な場合）
- 可能な範囲での停止保持

MRC失敗事象は本書の安全論証では「設計上発生しないとは主張できない残存事象」として扱い、Phase 1〜3の運用エビデンス蓄積に基づき、発生確率を許容残存リスク水準まで管理する（第21章、第2版TOC）。MRC失敗事象のログ記録、事故時のデータ提出プロセスは第13章で扱う。

# 12. 遠隔監視と運用安全

## 12.1 遠隔監視の位置づけ

遠隔監視は、車両単独MRCを補完する運用手段である。遠隔監視は、MRC成立の一次安全手段ではない。

TuringのL4安全論証では、車両が遠隔監視に依存せず、故障検知、フェイルオーバー、MRC、停止保持を実行できることを前提とする（第9章、第11章）。遠隔監視は、(i) MRC後の状態確認・乗客対応・現場対応の運用支援、(ii) ODD境界・運用条件の運用判断支援、(iii) 例外事象（事故、インシデント、警察・消防対応等）への運用対応のために用いる。

第2版では、遠隔監視を、日本の道路交通法における**特定自動運行**制度およびその主任者（特定自動運行主任者）の責任構造と整合させて扱う。具体的には、特定自動運行主任者は遠隔監視装置を介して、特定自動運行用自動車の運行を監視し、必要に応じて警察への通報、消防への通報、現場対応の手配等を行う主体である。本書本体では遠隔監視の安全論証上の位置づけを示し、特定自動運行制度との接続は第12.5節で扱う。

遠隔監視装置の具体的な要件、オペレータの資格・教育、シフト体制、監視映像の品質、通信構成、ヒューマンファクタ要件は、衛星文書「遠隔監視運用設計書」「特定自動運行計画書」で確定する。

## 12.2 遠隔監視をMRC成立根拠にしない理由

通信断、遠隔監視システム障害、オペレータ不在、通信遅延等が発生しても、車両は単独でMRCを完了する必要がある。

理由は以下である。

- 遠隔監視通信は、無線通信品質、ネットワーク輻輳、サービス障害、車両側通信ハードウェア故障により、利用可能性が一定でない
- 遠隔オペレータの認知・判断・操作には人間特性に起因する遅延がある
- 緊急時にリアルタイム操舵・制動を遠隔オペレータに依存すると、ヒューマンファクタ起因のリスクと、オペレータが同時監視可能な車両台数に応じた運用コストが大きくなる
- UN-R 157のFailsafe Response（5.4）およびMRM（5.1.5）の考え方は、車両自身の能力でMRCを成立させることを基本とする

したがって、遠隔監視をリアルタイム操舵・制動の代替手段として扱わない。遠隔監視に期待しないことを設計前提とすることで、MRC capabilityの安全論証を通信品質に依存させない。

## 12.3 MRC後の遠隔監視の役割

遠隔監視は、MRC後に以下を支援する。

| **領域** | **役割** | **対応する制度・基準** |
| --- | --- | --- |
| 停止後状態確認 | 車両位置、周囲状況、乗員状態を確認する | 道路交通法第75条の12・14（特定自動運行主任者の遠隔監視義務） |
| 乗客対応 | 車内通話、案内、降車判断支援を行う | 同上、運送事業者として |
| 周囲対応 | 必要に応じて関係者・所有者・通行人等へ連絡する | 同上 |
| 警察・消防への通報 | 事故・故障・周囲交通への影響発生時に通報する | 道路交通法第75条の14（事故時の通報義務） |
| 現場対応 | 駆けつけ、車両回収、再配置を行う | 道路交通法施行令の規定する措置 |
| 記録保全 | MRCログ、映像、車両状態、通信記録を保全する | 第13章、UN-R 157 DSSAD相当 |

各役割の具体的な手順、所要時間目標、SLA、関係者間連携は、衛星文書「遠隔監視運用設計書」「特定自動運行計画書」「事故・インシデント対応プロセス（第2版TOC第18章）」で扱う。

## 12.4 遠隔監視断時の扱い

遠隔監視との通信が断たれた場合でも、車両は通常走行を無条件に継続しない。

遠隔監視断がODDまたは運用条件上の逸脱に該当する場合には、安全側に判断し、必要に応じてMRCへ遷移する。判定主体は安全監視系（第9.5節）である。判定の入力は通信品質監視（第10.10節）と運用条件である。

ただし、MRCそのものは遠隔監視に依存せず、車両単独で完了する（第12.2節）。MRC完了後の状態確認・乗客対応・現場対応は、通信が回復した時点で遠隔監視が引き継ぐか、回復しない場合は事前に定めた現場対応手順に従う。

遠隔監視断時の通常走行継続可否判定の定量基準（許容断時間、許容劣化レベル等）、運用条件としての断時許容性、現場対応への引継条件は、衛星文書「遠隔監視運用設計書」「特定自動運行計画書」で確定する。

# 13. ログ・記録保全

## 13.1 ログを安全論証に含める理由

L4自動運転車両では、運転者が車両挙動を説明する主体ではない。

そのため、車両がどのような入力を受け、どのような出力を生成し、どのような状態遷移を行い、どのように停止したかを、事後に再現可能な形で記録する必要がある。ログは、安全論証、事故調査、改善、モデル更新、運用改善の基礎となる。

第2版では、本章のログ要件を、UN-R 157のDSSAD（Data Storage System for Automated Driving）要件と対応付けて整理する（第6.10節）。形式的準拠は目的としない（第6.1節）が、対外説明・協議の共通言語としてDSSAD要件を参照する。記録項目の詳細仕様は衛星文書「ログ項目定義書」で確定し、本書本体では分類・対応関係・運用上の位置づけを示す。

## 13.2 記録すべき主な情報

本書では、少なくとも以下を記録対象として扱う。各分類はUN-R 157 DSSAD要件と対応付けて整理する。

| **分類** | **記録内容** | **UN-R 157 DSSAD対応項目** |
| --- | --- | --- |
| E2E AI入力 | カメラ画像、過去フレーム、車両状態量、入力品質指標 | データ要素（システム入力） |
| E2E AI出力 | 舵角指示、加速度指示、補助出力、推論時刻、信頼度・不確実性 | データ要素（システム出力） |
| 安全監視 | 入力品質判定、出力異常検知、ODD逸脱、車両応答、判定の根拠 | システム失陥、ODD逸脱事象 |
| 通常走行ADS | 通常走行状態、異常、停止、遅延、ハートビート | システム作動状態、システム失陥 |
| MRC系ADS | 起動、フェイルオーバー、MRCモード遷移、MRC専用E2E AIの入出力 | MRM（Minimum Risk Manoeuvre）作動 |
| フェイルオーバー | 開始時刻、理由、権限移譲、完了状態、Arbitration判定 | 状態遷移、Override |
| MRC | 開始条件、制御指令、停止位置選定、停止結果、完了状態 | MRM作動、MRM完了 |
| 車両制御 | 制動、操舵、駆動遮断、ハザード、外向けHMI、車両側Plausibility Check結果 | データ要素（車両制御） |
| 車両状態量 | 車速、操舵角、加速度、位置、ヨーレート、タイヤ空気圧、電源状態 | データ要素（車両状態） |
| 遠隔監視 | 通信状態、オペレータ対応、現場対応、特定自動運行主任者の通報・指示 | Override、運用記録 |
| ODD関連 | ODD内/境界/外判定、Triggering Condition検知、ジオフェンス判定 | ODD逸脱事象、システム作動条件 |
| AI不確実性 | E2E AIの不確実性申告、OOD検知、AI Safety Requirements関連監視 | データ要素（任意拡張） |
| サイバーセキュリティ | 重大セキュリティイベント、認証異常、改ざん検知 | UN-R 155 CSMSと連携 |
| OTA / 更新 | モデル更新、ソフトウェア更新、ロールバック、rolling update中の系間バージョン | UN-R 156 SUMSと連携 |

各記録項目の保存期間、保存形式、保存場所（車載記録 / クラウドアップロード）、暗号化、改ざん防止、アクセス制御は、衛星文書「ログ項目定義書」「データ管理方針」で確定する。本書本体ではUN-R 157 6項のData elements / Activation / Transition / Override / MRM / System failure / Storageの観点参照を維持する（第6.10節）。

## 13.3 事後再現性の考え方

ログは、単なるイベント記録ではなく、事後に車両挙動を再現できることを目指す。そのため、以下を重視する。

| 重視する項目 | 内容 |
| --- | --- |
| 時刻同期 | 全ECUでPTP同期。タイムスタンプはUTCおよび車載時刻の両方を保持 |
| イベント順序 | 因果関係を再構成可能な順序保証。複数ECU間の事象でも順序が一意に定まる |
| E2E AI入力と出力の対応 | フレーム単位で入力画像と出力指令を紐付け可能とする |
| 安全監視判断の根拠 | 安全監視系がどの観測値からどの判定に至ったかを再構成可能とする |
| フェイルオーバーの理由 | フェイルオーバー直前の通常走行ADS状態、安全監視判定、Arbitration入力を保存する |
| MRC遷移の理由 | MRC開始条件（第11.2節）のいずれが成立したかを記録する |
| MRC位置選定の根拠 | 第11.4.3節のMRC位置選定アルゴリズムの入力・出力・判定経路を記録する |
| 車両制御指令と実車応答の対応 | 制御指令と車両応答（実車速、実加速度等）の差を再構成可能とする |
| 停止結果 | 停止位置、停止姿勢、停止後の保持状態 |
| 遠隔監視対応 | 通信状態の遷移、オペレータの介入記録、特定自動運行主任者の通報・指示 |

事後再現性の検証は、衛星文書「検証文書群」「事故再現プロセス書」で扱う。ログから再現可能な範囲、再現精度の目標、再現できない事象の扱いも同文書群で確定する。

## 13.4 事故時のデータ提出プロセス

第2版では、事故・重大インシデント発生時のデータ提出プロセスを独立節として扱う。本節は本書のL4安全論証における事故時データ取扱いの位置づけを示すものであり、具体的な手続・SLA・関係者連携は衛星文書「事故・インシデント対応プロセス（第2版TOC第18章）」「ログ項目定義書」「特定自動運行計画書」で確定する。

### 13.4.1 提出対象

事故・重大インシデント発生時に提出対象となるデータは、以下を含む。

| 提出対象 | 提出先 | 根拠 |
| --- | --- | --- |
| DSSAD相当ログ（事故前後の所定時間幅） | 警察、運輸安全委員会、国土交通省、NALTEC、公安委員会 | 道路交通法、道路運送車両法、UN-R 157相当 |
| カメラ映像（事故前後の所定時間幅） | 同上 | 同上 |
| E2E AI入出力（事故前後の所定時間幅） | 同上、必要に応じてOEM・サプライヤ | 安全論証エビデンス |
| 車両状態量（事故前後の所定時間幅） | 同上 | 同上 |
| MRC関連ログ（MRC関与事故の場合） | 同上 | 同上 |
| 遠隔監視・通信記録 | 同上、運用主体 | 特定自動運行制度関連 |
| ソフトウェア・モデルバージョン情報、変更履歴 | 同上、必要に応じてOEM・サプライヤ | UN-R 156、変更管理 |
| 関連する設計文書・安全論証文書 | 当局・調査機関の要請に応じて | 法令、調査協力 |

提出対象データの所定時間幅、解像度、形式、暗号化解除手順は、衛星文書「ログ項目定義書」で確定する。

### 13.4.2 提出プロセスの基本方針

事故時のデータ提出は、以下の基本方針に従う。

- データの**改ざん防止**：事故発生時点までに記録されたログの完全性を保証する。改ざん検知メカニズム（電子署名、ハッシュチェーン等）を実装する。
- データの**保全**：事故後にデータが上書き・削除されないよう、事故検知時点で対象データを保護領域に隔離する。
- データの**アクセス制御**：提出先の認可された主体のみがデータにアクセスできるよう、認証・認可・ログを実装する。
- **時間SLA**：法令に定める通報義務の時間枠（道路交通法第75条の14）を満たす提出体制を維持する。
- **関係者連携**：警察・運輸安全委員会・国土交通省・NALTEC・公安委員会・運用主体・OEM・サプライヤ・Turing間の連絡体制を事前に整備する。
- **被害者・遺族への配慮**：個人情報・プライバシーに配慮しつつ、必要な情報開示を行う。
- **業界全体への情報共有**：再発防止に資する範囲で、匿名化された知見を業界へ共有する（第19章、第2版TOC）。

### 13.4.3 サイバーセキュリティ・プライバシーとの整合

事故時データ提出にあたっては、サイバーセキュリティ（第14章、第2版TOC）およびプライバシー保護（個人情報保護法、車内・車外被撮影者の権利）との整合を確保する。具体的な取扱いは、衛星文書「Cybersecurity Case」「データ管理方針」「事故・インシデント対応プロセス」で扱う。

### 13.4.4 Phase別の運用方針

事故時データ提出プロセスは、Phase 1（実証フェーズ）、Phase 2（限定商用フェーズ）、Phase 3（本格商用フェーズ）で段階的に成熟させる。各Phaseの運用方針は、衛星文書「特定自動運行計画書」「事故・インシデント対応プロセス」で確定する。本書本体ではプロセスが各Phaseで運用可能な状態にあることを安全論証の前提とする。

# 17. 結論

## 17.1 TuringのL4安全論証の要点

TuringのL4安全論証の要点は、以下である。

第一に、Turingの通常走行ADSは、認識・予測・判断・制御を個別モジュールとして分ける従来型ADSではなく、画像から舵角指示および加速度指示を出力するE2E AIを中核とする。

第二に、E2E AIの安全性は、内部モジュールの正しさではなく、入力画像に対する出力指示と、その結果としての閉ループ車両挙動として論証する。

第三に、ハードウェア失陥は、E2E AIの通常機能で吸収しない。カメラ、SoC、ECU、通信、電源、制動、操舵等の失陥は、多重系ADS、安全監視、フェイルオーバー、MRCによって扱う。

第四に、多重系ADSの目的は、一系統失陥後の通常走行継続ではなく、MRC系ADSによるMRC達成である。フェイルオーバー後は通常走行へ自動復帰せず、MRCモードに遷移する。

第五に、遠隔監視はMRC成立の一次安全手段ではない。車両は、通信断や遠隔監視断があっても、単独でMRCへ移行できる必要がある。

第六に、ログ・記録保全は、安全論証、事後説明、改善、モデル更新の基盤である。

## 17.2 最終主張

TuringのL4自動運転車両は、定義されたODD内において、E2E AIにより通常走行時の安全な閉ループ車両挙動を実現することを目指す。

同時に、Turingは、E2E AIを用いることをもって故障時安全までAI任せにするものではない。

ADS、センサ、計算機、通信、電源、制動、操舵等に失陥が生じた場合には、多重系ADS、安全監視、車両制御I/Fにより、通常走行を継続せずMRCモードへ遷移し、車両単独でリスク最小状態に到達する。

この構造により、Turingは、E2E AIの新規性と、L4車両に求められる失陥時安全を両立する安全論証を構成する。
